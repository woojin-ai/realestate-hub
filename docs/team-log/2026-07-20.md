1. 08-01 01:00 KST 크론(=07-31T16:00Z): `getYmList` index0 = **202607**(UTC는 아직 7월) → 수집·`ready` 마킹

### 🎯 최우선 항목이던 "부분월 건수 감소" — 원인 규명 + 수정 완료 (`f3fdeb8`)

**원인 — 같은 달을 두 경로로 서빙하는데 세는 기준이 달랐다.**
- 캐시 경로(`loadMonthFromDb`) = `deals` 테이블 = 자연키로 **이미 접힌** 행

`app/api/data/route.ts`의 최신월(index 0)만 캐시 신선도에 따라 두 경로를 오간다. 그래서 사용자에겐 진행 중인 달 건수가 조회할 때마다 오르내리는 것처럼 보였고, **완결월이 항상 안정적이었던 것도 이걸로 설명된다**(완결월은 항상 캐시 경로).

07-19에 관측된 수치가 정확히 이 구조로 재현됐다:
```
국토부 원본 raw   전세 270 / 매매 133   ← 01:29 조회 = 라이브 경로
자연키 distinct   전세 258 / 매매 130   ← 01:45 조회 = 캐시 경로
```

**부수 발견 — 매일 09:00 KST에 재발하는 구조였다.** `route.ts:96`의 `new Date().toISOString().slice(0,10)`은 **UTC 날짜**라 "오늘" 경계가 09:00 KST다. 그 직후 첫 조회가 라이브로 떨어져 건수가 올라갔다가, 그 조회가 캐시를 `ready`로 마킹한 뒤부터 다시 내려갔다. 1회성 이상이 아니라 **매일 반복되던 현상**이다.

### QA 핑퐁 — FAIL 4건 적발 → 교정 → 재검증 전부 PASS

**1차 검증에서 FAIL 2건 + 누락 진입 경로 2곳**이 나왔다. 전부 마스터가 직접 코드로 확인했고 **4건 모두 타당했다.**

1. **(주석) "라이브 사이트 수치 변동은 없다"가 사실과 다름** — 실제로는 최신월이 270→258로 내려간다. **이건 개발팀 잘못이 아니라 마스터 지시 문구가 부정확했던 탓이다** — 마스터가 "표시 수치 불변"이라 지시했고("최종적으로 사용자가 보는 값은 같다"는 뜻이었다) 개발팀이 그대로 주석에 옮겼다. 정확한 서술로 교체.
2. **(주석) `contractTerm` 서술 부정확** — `contract_type`과 나란히 "키에서 빠져 있다"고 썼으나 성격이 전혀 다르다. `contract_type`은 파싱되어 `RentRecord`·`DealRow`·`deals` 컬럼에 **모두 있고 키에만** 없다(키 확장으로 해결 가능). `contractTerm`은 **파서에 아예 없다**(`grep lib/molit-api.ts` → 0건). "`dealNaturalKey` 한 곳만 고치면 된다"는 문장을 그대로 뒀으면 후속 작업자가 막다른 길로 갔다.
