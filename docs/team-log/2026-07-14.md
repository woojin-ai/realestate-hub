## 추가 라운드: 마이그레이션 3단계(캐시/영속화) — Supabase 연동

### 목표
Supabase 프로젝트가 준비되어(`.env.local` 3개 키 채워짐, deals 등 스키마 이미 적용됨) 로드맵 3단계 진행.
`app/api/data/route.ts`가 국토부 API를 매번 직접 호출하던 것을, Supabase `deals`/`fetch_cache_status`/
`monthly_stats` 테이블을 우선 조회하고 없거나 오래된 개월만 국토부 API로 채우는 방식으로 변경.

### 시작 전 확인
- `.claude/agents/dev-team.md`, `docs/planning/migration-architecture.md`(3단계 섹션·서버리스 제약 대응),
  `supabase/schema.sql`, 기존 `app/api/data/route.ts`/`lib/molit-api.ts`/`lib/analyzer.ts`,
  `docs/design/dashboard.md`(fetch 상태 UX) 모두 Read.
- `node_modules/next/dist/docs/` 재확인 — 여전히 존재하지 않음(표준 Next.js 16.2.10). 기존 표준 컨벤션 유지.
- **동시 작업 발견**: 작업 시작 시점에 `lib/molit-api.ts`, `lib/analyzer.ts`, `app/page.tsx`,
  `components/BuildingTypeToggle.tsx`가 이미 마지막 커밋 대비 수정된 상태(아파트 외 빌라/단독 API 호출
  일반화 등, 이번 캐시 작업과 무관한 내용)로 되어 있었음 — 다른 프로세스가 동시에 작업 중인 것으로 보임.
  이 파일들은 이번 라운드 범위가 아니므로 손대지 않았고, 커밋 시에도 포함하지 않음. 다만 `app/api/data/route.ts`
  는 이미 그 일반화(building_type 파라미터)가 반영된 상태였고 이번 캐시 로직을 그 위에 얹었기 때문에,
  이 파일을 커밋하면 그 building_type 일반화 변경분도 함께 포함된다(파일 단위 스테이징 한계).
  마스터 확인 필요: 해당 building_type 일반화 작업의 담당/검증 주체를 확인해달라.

### 구현 내역
1. `lib/supabase.ts`(신규) — 서버 전용 Supabase 클라이언트(`SUPABASE_SERVICE_ROLE_KEY`로 RLS 우회).
   `@supabase/supabase-js` 신규 설치(`package.json`/`package-lock.json` 갱신).
2. `lib/db-cache.ts`(신규) — Supabase read/write 헬퍼.
   - `upsertMonthDeals`: 한 달치 `MonthData`를 `deals` 테이블에 upsert. 자연키(unique 제약)에
     price/deposit/monthly가 모두 포함되는데, 해당 거래유형에 안 쓰이는 필드를 NULL로 두면
     Postgres가 NULL끼리는 서로 다른 값으로 취급해 재적재 시 중복이 쌓이는 문제가 있어 0으로
     채움(analyzer.ts가 이미 `price>0`/`deposit>0`을 "값 없음" 기준으로 쓰고 있어 의미 충돌 없음).
   - **버그 발견 및 수정**: 국토부 API가 동일 자연키의 거래를 같은 달에 중복 반환하는 경우가 실제로
     있어(`ON CONFLICT DO UPDATE command cannot affect row a second time` Postgres 에러 재현됨),
     upsert 배치 전송 전 자연키 기준 중복 제거(`dedupeByNaturalKey`) 로직 추가.
   - `loadMonthFromDb`: 저장된 한 달치를 다시 읽어 `TradeRecord`/`RentRecord` 형태로 복원
     (PostgREST 1000행 제한 대비 페이지네이션 포함).
   - `getCacheStatus`/`upsertCacheStatus`: `fetch_cache_status` 조회/갱신.
   - `upsertMonthlyStats`: `analyzer.ts`의 `buildMonthlyStats` 결과를 `monthly_stats`에 반영.
3. `app/api/data/route.ts` 수정 — 요청 시 `fetch_cache_status`를 먼저 조회해, 이미 수집된 개월
   범위(`months_collected`) 안에 있는 과거 개월은 DB에서 읽고, 최신월(index 0)은 오늘 이미 수집을
   마친 경우(`last_fetched_at`이 오늘)에만 DB를 재사용하며, 그 외(신규 지역/부족한 개월/최신월 갱신
   필요)는 국토부 API를 호출해 `deals`에 upsert 후 `monthly_stats`/`fetch_cache_status` 갱신.
   Supabase 초기화/조회/쓰기가 실패해도 전부 try/catch로 감싸 국토부 API 직접 호출로 완전히
   폴백(캐시는 성능/영속화 부가기능일 뿐 조회 자체를 막지 않음).
4. `lib/types.ts`에 `FetchSource`("cache"|"live"|"partial") 타입과 `DealsApiResponse.source` 필드
   추가. `docs/design/dashboard.md`의 `FetchStatusBanner` 상태 스펙(cache-hit/collecting/ready 등)은
   있으나 현재 화면(`app/page.tsx`)은 그 배너를 아직 구현하지 않은 단순 idle/loading/ready/error 상태만
   쓰고 있어(새 화면을 임의로 설계하지 않기 위해), `source` 필드만 최소 추가하고 화면 컴포넌트는 건드리지
   않음 — 화면에서 실제로 캐시 상태를 사용자에게 보여주려면 디자인팀 구성안대로 `FetchStatusBanner`
   컴포넌트 구현이 필요(범위 밖, 마스터 확인 요망).
