## 0. 원본 동작 요약 (사실 확인)

`recommender.py score_apartments()` + `app.py /api/recommend`의 실제 동작:

- **종합점수 = 가격 40% + 역세권 30% + 신축 20% + 평지 10%** (가중치는 슬라이더로 조정, 합 100%).
- **입력 파라미터**: `lawd_cd`, `gu`, `budget`, `budget_type`(`max`=이하/`min`=이상), `deal_type`(매매/전세), `year`(신축기준연도), `households`(최소 세대수), `w_price`/`w_subway`/`w_new`/`w_slope`.
- **후보 풀**: `build_apt_stats(..., top_n=200)`로 단지 최대 200개를 뽑고, **예산·신축·세대수로 먼저 필터링**한 뒤 남은 후보만 스코어링.
- **단지별 외부 호출(순차 + sleep)**:
  1. `get_coordinates()` — 카카오 키워드검색 1회 → 좌표 (`+ sleep 0.1s`)
  2. `get_nearest_subway()` — 카카오 카테고리검색(SW8) 1회 → 최근접 지하철 거리 (`+ sleep 0.1s`)
  3. `get_slope_score_vworld()` — opentopodata `srtm30m`에 5개 점 배치 1회(실패 시 open-elevation로 폴백) → 고도범위로 평지점수
  4. (세대수 필터 사용 시) `_get_kapt_code()` + 세대수 캐시 조회 — 추가 API 경로
- **점수 산식**:
  - 가격점수 = `(지역 max_price − 단지 avg_price) / (max−min) × 100` (저렴할수록 고득점)
  - 역세권점수 = 거리 구간별 (≤300m 100 / ≤500m 85 / ≤800m 65 / ≤1200m 40 / 그외 15)
  - 신축점수 = 기준연도 구간별 (기준≤ 100 / −5년 70 / −10년 40 / 그외 10). `year=0`이면 절대연도(2026 기준) 구간 사용.
  - 평지점수 = 고도범위 (≤5m 90 / ≤15m 70 / ≤30m 40 / 그외 15), 데이터부족 시 50.
- **컷오프**: 예산 초과/미달 제외, 신축점수<40 제외, 최종 **상위 10개** 반환.
- **전송 방식**: SSE(`EventSource`)로 `progress`(현재/전체/%/단지명) 이벤트를 흘리다가 `done`으로 결과 배열 전송. 1~2분 걸리는 연산을 진행바로 버티게 한 구조.
- **카드 UI(889~943줄)**: 랭킹 메달(🥇🥈🥉/N위), 종합점수 `NN점/100`, 점수 막대 4종(가격 빨강·역세권 파랑·신축 초록·평지 주황), 지하철 배지(`🚇 역명 · 도보 Nm`), 평수별 시세 표(예산내 행 초록 하이라이트), 카드 클릭 시 평수상세 모달(`openAreaModalFromRec`) 연결.

신 프로젝트 재사용 자산: `lib/analyzer.ts buildAptStats(allData, dealType, topN)`가 `AptStat{name, dong, build_year, avg_price, count, area_stats...}` 배열을 이미 생성 → **추천 입력으로 그대로 재사용 가능**. `.env.local`에 `KAKAO_API_KEY`, `VWORLD_API_KEY`, `MOLIT_API_KEY` 채워져 있음.

> ⚠️ 주의: 원본은 `build_apt_stats(top_n=200)`을 쓰지만 현 `lib/analyzer.ts buildAptStats`의 **기본 `topN=30`**이다. 추천용으로는 dev 단계에서 `topN`을 100~200으로 넉넉히 넘겨 후보 풀을 확보해야 한다(시그니처는 이미 지원).

## 1. 기능 범위 정의 (MVP vs 풀버전)

핵심 판단: **평지(고도)와 세대수 필터가 서버리스에서 가장 위험/불안정한 요소**다. 이 둘을 MVP에서 빼면 카카오 호출만으로 완결되어 타임아웃/외부 의존 리스크가 크게 준다.

### MVP (1차 릴리스) — "가격·역세권·신축 3요소"
- **점수 = 가격 + 역세권 + 신축** (평지 제외). 가중치는 3요소 합 100%로 재정규화(예: 기본 45/35/20).
  - 대안: 평지점수를 UI에는 남기되 항상 중립값(50점)으로 고정 표기하고 "정밀 지형분석 준비중" 라벨 → **비추천**(오해 소지). MVP는 평지 항목 자체를 카드에서 빼는 쪽을 권고.
- **사용자 입력**: 예산 + 예산유형(이하/이상), 신축기준연도, 거래유형(대시보드 현재 탭과 연동), 가중치 슬라이더 3종(가격/역세권/신축).
- **세대수 필터 제외**(추가 kapt API 경로가 무거움 → 2차).
- **외부 호출**: 단지당 카카오 2회(좌표+지하철)뿐.

### 풀버전 (2차 이후)
- 평지(고도) 점수 복원 — 단, **좌표별 고도 결과를 영속 캐시**하고 opentopodata 쿼터/안정성 확보를 전제로.
- 세대수 필터(`_get_kapt_code` + 세대수 캐시 이관).
- 카드 클릭 → 평수상세 모달, 주변시설(`/api/nearby`: 학교/병원/편의시설) 연계.

## 2. 서버리스 타임아웃 대응 — 설계 옵션 비교

문제: 원본은 후보 ~30개 × (카카오 2~3회 + 고도 1회) 순차 + sleep = **1~2분**. Vercel 서버리스 함수는 실행시간 상한(플랜별 10~60초, `migration-architecture.md` 서버리스 제약 섹션)이 있어 그대로 이식하면 **콜드 실행에서 타임아웃 확정**.

| 옵션 | 내용 | 장점 | 단점 |
|---|---|---|---|
| (a) 후보 수 축소 | 상위 N개(예: 8~12개)만 스코어링 | 구현 단순, 단일 호출로 끝 | 커버리지 좁음. 가중치 슬라이더 재조정마다 카카오 재호출(캐시 없으면 비용/시간 반복). 사용자가 "왜 이 단지만?" 의문 |
| (b) 카카오 결과 영속 캐시(Supabase) | 좌표·최근접지하철을 단지 키로 DB 저장. 좌표/역거리는 **거의 불변** → 영속 캐시에 최적 | 두 번째 조회부터 외부호출 0에 수렴 → 사실상 즉시 응답. 쿼터/타임아웃 동시 해결. 여러 사용자가 캐시 공유 | 최초(콜드) 채우기 순간은 여전히 시간 소요 → (a)/(d)와 결합 필요. 캐시 테이블/무효화 정책 필요 |
| (c) 스트리밍(SSE/streaming response) | 원본처럼 진행상황 흘리며 점진 응답 | UX(진행바) 유지, 부분결과 조기표시 | **총 실행시간 상한 자체는 못 늘림** — 연결을 열어둬도 함수 max duration에 걸리면 끊김. 근본 해결 아님 |
| (d) 배치 분할 호출 | 클라이언트가 API를 여러 번(배치 N개씩) 호출해 누적 | 각 호출이 상한 내로 짧음, 상한 우회 | 클라이언트 오케스트레이션 복잡, 부분 실패 처리, 중간상태 관리 필요 |

### ✅ 추천안: **(b) 영속 캐시를 토대로, 콜드스타트는 (a) 후보 상한 + (d) 소규모 배치로 보강**

구체 설계:
1. **`apt_geo` 캐시 테이블(Supabase)** 신설 — 키 `(lawd_cd, name, dong)`, 값 `lat, lng, subway_name, subway_dist, geocoded_at`. 좌표·역거리는 단지 특성상 불변에 가까우므로 **만료 없는 영속 캐시**(필요 시 수동/주기 재검증).
2. 추천 요청 시: `buildAptStats`로 후보 필터 → 각 후보를 `apt_geo`에서 조회.
   - **캐시 히트**: 외부호출 0. 즉시 점수 계산.
   - **캐시 미스**: 이번 호출에서 최대 `K개`(예: 10~12개)만 카카오 호출로 채우고 DB 저장. 상한 초과분은 이번 라운드 점수에서 제외하거나 다음 호출로 이월.
3. **첫 조회 UX**: "지역 데이터 준비 중" 배지 + 부분결과 즉시 표시 → 재요청/배경 채움으로 점진 완성(마이그레이션 아키텍처의 "부분 데이터 즉시 표시 후 점진 갱신" 원칙과 동일).
4. **가중치 재조정은 외부호출 없이 즉시** — 점수 재계산은 캐시된 geo만으로 클라이언트/서버에서 수행. 슬라이더가 카카오를 다시 때리지 않는 게 핵심 이득.
5. 평지(고도)는 **별도 캐시 컬럼(`slope_score`)** 으로 풀버전에서 붙이되, opentopodata 쿼터 때문에 콜드 채움을 Cron 배경작업으로 분리 검토.

> 근거: 원본의 병목은 "매 요청 실시간 외부호출"이다. 좌표/역거리는 변하지 않는 데이터이므로 **한 번 구해 DB에 박아두면** 서버리스 상한과 쿼터 문제가 동시에 사라진다. (a)만 쓰면 커버리지가 좁고 슬라이더 재호출 비용이 남으며, (c)는 상한을 못 늘려 근본 해결이 아니다.

## 3. API 사용량 / 쿼터 리스크

### 카카오 REST (로컬 검색)
- 호출 추정: **단지당 2회**(키워드검색 좌표 1 + 카테고리 SW8 1). 후보 30개 콜드 = **60회**. (풀버전 세대수/주변시설 붙으면 단지당 3~6회로 증가)
- 캐시 효과: 같은 지역 재조회·가중치 재조정 시 **0에 수렴**. 신규 지역·신규 단지 등장분만 증분 호출.
- 리스크: 카카오 로컬 API 일일 쿼터 한도는 **확인 필요 항목**(§6). 다만 60회/지역 수준은 어떤 합리적 쿼터에서도 문제 없고, 영속 캐시로 반복 호출이 사라지므로 실질 리스크는 낮음.

### 고도 API (풀버전)
- opentopodata 공개 엔드포인트는 **rate limit(대략 초당 1회·일 1000회급)** 및 가용성 불안정이 알려져 있음. open-elevation 폴백도 간헐 장애.
- 후보 30개 = 고도 배치 30회. 여러 지역/사용자 누적 시 일일 한도에 근접 가능 → **반드시 결과를 영속 캐시**하고 콜드 채움을 배경 Cron으로 분리. 이 불안정성이 평지를 MVP에서 뺀 핵심 이유.

## 4. 화면 위치 / 진입점

### 권고: **대시보드 조회 결과 아래 "AI 맞춤 추천" 섹션**(원본 구조 계승) + 클라이언트 트리거
- 추천은 사용자가 **이미 특정 지역을 조회한 컨텍스트** 위에서 동작한다(`lawd_cd`, `gu`, 현재 거래탭 재사용). 별도 페이지로 떼면 지역 선택을 다시 시키는 마찰이 생김.
- 접힘(collapsed) 섹션 → "맞춤 추천 받기" 버튼 클릭 시 입력폼(예산/연도/가중치) 펼침 → 실행. 초기 로드시엔 호출 안 함(불필요한 API 절약).

### SEO 관점
- 추천 결과는 **사용자 입력 의존·개인화·동적** 콘텐츠 → 검색 인덱스 가치 낮고, 얇은/중복 페이지로 취급될 위험.
- 따라서 **독립 인덱싱 페이지로 만들지 않는다.** 대시보드 내 클라이언트 섹션으로 두고, 만약 `/recommend` 라우트를 별도로 판다면 `noindex` 처리.
- SEO 트래픽은 지역별 시세 대시보드(정적/ISR 가능한 조회 결과) 쪽에서 확보하고, 추천은 체류시간·재방문(engagement) 기여로 포지셔닝.

## 6. 미결 / 사용자(마스터) 확인 필요 항목

1. **카카오 로컬 API 일일 쿼터 한도 확인** — 캐시로 반복 호출은 없앴으나, 신규 지역 대량 콜드 채움 시 상한 확인 필요.
2. **Vercel 플랜 함수 max duration 확정** — Hobby/Pro에 따라 후보 상한 `K`와 배치 크기 튜닝이 달라짐. 현재 배포 플랜 확인 필요.
3. **고도(opentopodata/open-elevation) 안정성·쿼터** — 평지 점수 복원(풀버전) 가부의 핵심 전제. 불안정하면 평지 항목을 영구 제외하거나 유료/대체 고도 소스 검토.
