# 다음 화면/기능 제안 (전국 데이터 완주 이후) — 2026-07-17

작성: 기획팀 / 상태: **기획 초안 (구현·배포 전, 마스터 검토용)**

## 0. 배경 — 지금 우리가 가진 데이터 자산

전국 114개 시·군·구 아파트 실거래가 프리워밍 1차 완주(라운드6). 이 시점의 핵심은
**"조회한 지역 하나"가 아니라 "전국이 이미 DB에 상주"** 한다는 점이다. 특히:

- **`monthly_stats`** (lawd_cd × building_type × deal_type × deal_ym → avg_price, deal_count)
  - 전국 114개 시군구 × 아파트 × 매매/전세 × 최대 13개월 ≈ 3,000행 남짓.
  - `public read` RLS 켜져 있어 **서버/클라 어디서든 Supabase 직접 조회 가능**.
  - **국토부 API 콜이 0건** — 이 테이블만 읽는 기능은 레이트리밋/콜예산과 완전히 무관.
- **`deals`** 원자료: 단지명 인덱스(`idx_deals_name`) + 지역·기간 인덱스 존재 → 단지별 13개월 추이 집계 가능.
- **`apt_geo`**: 지역별 단지 좌표/최근접 지하철(추천 기능이 채워온 캐시).
- 재사용 자산: `MonthlyChart`(chart.js 라인차트 패턴), `RegionSelector`, `SummaryCards`,
  `DealsTable`, `AptDetailModal`, `formatPrice/formatChange`(analyzer.ts), `REGION_CODES`.

> 핵심 통찰: 지금까지 화면은 전부 **"한 지역 드릴다운"** 이었다. 전국 데이터가 상주한 지금은
> **"전국을 가로로 비교/랭킹"** 하는 화면이 가장 큰 신규 가치이고, 그게 `monthly_stats`로
> **추가 API 콜 없이** 가능하다. 아래 우선순위는 이 관점을 최우선으로 반영했다.

## 1순위 — 전국 시세 랭킹 보드 `/ranking`

### 1) 무엇을 / 사용자 가치
- 전국 114개 시군구를 **평균 매매가 / 평균 전세가 / 전월 대비 상승률(하락률) / 거래량** 기준으로
  정렬해 보여주는 리그테이블 + 상승·하락 TOP10 하이라이트 페이지.
- 탭: 매매 / 전세. 정렬 토글: 평균가↑↓ · 상승률↑↓ · 거래량↑↓. 시/도 필터(예: 경기도 안에서만).
- 사용자 가치: "내 지역이 전국에서 몇 위인지", "지금 어디가 오르고 어디가 빠지는지"를
  **한 화면에서** 파악. 지금까지 불가능했던 "가로 비교" 수요를 정면으로 충족.
- 부가: 각 행 클릭 → 기존 메인 대시보드(`/?lawd_cd=...`)로 딥링크. 트래픽 순환 구조.

### 2) 왜 지금
- 전국 프리워밍 완주로 `monthly_stats`가 사실상 다 채워졌다. 랭킹은 **비교 대상 모수(전국)가
  갖춰져야** 의미가 있는데, 지금이 그 시점이다.
- 상승률/하락률은 `monthly_stats`의 최근 2개월 avg_price만 있으면 산출된다(analyzer의
  `pctChange` 로직 그대로 재사용 가능).

### 3) 필요한 데이터 / API
- **기존 자산만으로 가능.** 신규 국토부 수집 **불필요**.
- 신규 엔드포인트 `/api/ranking` (또는 서버 컴포넌트에서 Supabase 직접 조회):
  `monthly_stats`에서 `building_type='아파트'`, 최근 2개 `deal_ym` 행을 전량 read → 지역별로
  현재월 avg + 전월 대비 변화율 계산 후 정렬.
- **콜예산 영향: 없음(molit 0콜).** Supabase read 한 번(≈3,000행, 인덱스/PK 커버). 부하 미미.
- **ISR 권장**: 데이터는 하루 1회 프리워밍 주기로만 바뀌므로 `revalidate`(예: 1~6시간)로
  캐싱하면 Supabase read도 사실상 방문당이 아니라 주기당으로 수렴. SSG 사이트 성격과 정합.

### 4) 난이도 / 리스크
- **난이도: 중.** 신규 라우트 1개 + 신규 페이지 1개 + 정렬/탭 UI. 차트는 선택(막대/히트맵).
  기존 `formatPrice`, `pctChange`, 색상 토큰 재사용 가능.
- 리스크
  - **YMYL: 낮음.** 공공데이터 사실(평균·건수·변화율) 표기이고 투자 권유/전망이 아님.
    단, "전국 1위 = 가장 좋은 동네"로 오해되지 않게 **중립적 카피**(단순 통계임을 명시).
  - **분석적 함정(중요):** 시군구 평균가는 **면적 구성에 크게 좌우**된다(대형 평형 비중 높은
    지역이 자동으로 상위). "평균가 랭킹 ≠ 평단가 랭킹"이라는 주석/툴팁을 반드시 넣어야
    오정보 논란을 피한다. (향후 `deals.area`로 평단가 랭킹 별도 제공 가능 — 확장 여지.)
  - **커버리지:** 대형 팬아웃 지역(화성 41590 등)이 아직 `collecting`이면 최신월이 비거나
    부분일 수 있다. 랭킹은 `deal_count`가 0이거나 avg가 null인 지역을 **"집계중"으로 별도 표기**
    (랭킹에서 제외 또는 하단 분리)해 빈 값이 1위처럼 보이는 오류를 막는다.
  - **성능: 낮은 리스크.** 데이터량 작고 ISR로 캐싱.
- **빌라/단독 제약: 저촉 없음.** 쿼리를 `building_type='아파트'`로 고정. UI에 유형 선택 노출 안 함.

### 왜 1순위인가
전국 데이터 완주로 **새로 가능해진 가치**를 가장 크게 실현하면서, **추가 콜예산 0**,
무료 플랜/레이트리밋 제약과 완전히 무관하고, SEO(신규 인덱싱 페이지)·체류시간·내부순환
트래픽까지 동시에 잡는다. 투입 대비 효과가 가장 높다.

## 2순위 — 관심지역 즐겨찾기 + 나란히 비교

### 1) 무엇을 / 사용자 가치
- 시군구를 **별표(★)로 즐겨찾기** 저장(로그인 없이 브라우저 `localStorage`).
- 상단에 즐겨찾기 칩, 클릭 시 즉시 조회. 나아가 **2~3개 지역을 나란히(side-by-side) 비교**
  (요약카드 + 미니 추이 라인 겹쳐보기).
- 사용자 가치: 이사·투자 검토 시 "관심 후보 몇 곳"을 반복 조회하는 실사용 패턴을 정면 지원.
  재방문 동기(리텐션) 강화.

### 2) 왜 지금
- 전국 데이터가 상주해 **어떤 조합을 즐겨찾아도 콜드스타트 없이 즉시** 뜬다(과거엔 비인기
  지역이 느렸을 것).
- `schema.sql`에 이미 **watchlist를 "로그인 생기기 전엔 localStorage로 대체"라고 설계 의도로
  주석**해 둠 → 그 방향의 자연스러운 첫 구현.

### 3) 필요한 데이터 / API
- **신규 수집 불필요.** 비교 요약은 기존 `/api/data` 재사용 또는 경량 신규 `/api/compare`
  (여러 lawd_cd의 `monthly_stats`만 묶어 반환, molit 0콜)로 처리.
- **콜예산 영향: 없음~미미.** 이미 프리워밍된 지역 재조회라 DB 캐시 히트.

### 4) 난이도 / 리스크
- **난이도: 하~중.** 핵심은 클라이언트 상태(localStorage) + 비교 레이아웃. 서버 변경 최소.
  `SummaryCards`·`MonthlyChart` 재사용, 비교 모드는 데이터셋 2~3개를 겹쳐 그리면 됨.
- 리스크
  - **성능/개인정보: 낮음.** localStorage는 서버 저장 없음.
  - SSR/hydration 주의(localStorage는 클라 전용) — 초기 렌더 mismatch 방지 처리 필요(경미).
- **빌라/단독 제약: 저촉 없음.** 아파트 building_type 고정 유지.

> 우선순위 근거: 가치·난이도 균형이 좋지만, 1순위(랭킹)가 만들어내는 "전국 비교" 맥락 위에
> 얹으면 시너지가 크다(랭킹에서 발견 → 즐겨찾기 → 비교). 그래서 랭킹 다음 순번.

## 3순위 — 단지별 시세 추이 차트 (AptDetailModal 확장)

### 1) 무엇을 / 사용자 가치
- 단지 상세(`AptDetailModal`)에 **해당 단지의 13개월 실거래 추이**(월별 평균/최고/최저 +
  거래건수 막대) 라인·복합 차트를 추가. 기존 `area_stats`(평형별)와 결합.
- 사용자 가치: 지금 단지표는 "최근 스냅샷"만 보여준다. 개별 단지의 **시간축 흐름**을 보여줘
  드릴다운의 깊이와 체류시간을 늘린다.

### 2) 왜 지금
- `deals` 원자료가 전국 13개월치 상주하고 `idx_deals_name`(lawd_cd, name) 인덱스가 있어
  단지 단위 집계가 가볍다. 스냅샷→추이로 확장할 재료가 이미 DB에 있다.

### 3) 필요한 데이터 / API
- **신규 수집 불필요.** 신규 `/api/apt-history?lawd_cd=&name=` : `deals`에서 단지명 필터로
  월별 집계(analyzer의 avg 로직 재사용). molit 0콜.
- **콜예산 영향: 없음.** 인덱스 적중 read.

### 4) 난이도 / 리스크
- **난이도: 중.** 신규 라우트 + 모달 내 차트 섹션. 표본 적은 달의 표시 규칙 설계 필요.
- 리스크
  - **YMYL/표본부족(주의):** 개별 단지는 월 1~2건 거래도 흔해, 추이선이 **한두 건에 요동**칠 수
    있다. "특정 자산의 시세를 단언"하는 인상을 주면 YMYL·오해 소지가 커진다 →
    **거래건수 동시 표기 + "표본 적은 달 주의" 안내 + 공공데이터 사실 표기 면책** 필수.
  - 성능: 단지별 쿼리라 지역 인기 시 호출 빈도↑ 가능하나, 인덱스+캐시로 관리 가능.
- **빌라/단독 제약: 저촉 없음.** deals 쿼리를 `building_type='아파트'`로 고정.

## 마스터에게 확인 요청 사항
1. **1순위(랭킹 보드) 우선 진행 승인** 여부 — 승인 시 다음 라운드에 디자인팀에 와이어프레임,
   개발팀에 `/api/ranking` + `/ranking` 페이지 스펙 전달.
2. 랭킹 지표 우선순위: 초기엔 **평균가 + 전월 대비 변화율 + 거래량** 3종으로 시작하는 것이
   맞는지(평단가 랭킹은 area 집계가 더 무거워 2차로 제안).
3. 이건 큰 방향 전환이 아니라 기존 대시보드 위 확장이라 결정 권한 범위로 보이나,
   **신규 최상위 페이지(`/ranking`) 추가**는 사이트 정보구조 변경이므로 사전 컨펌 요청.
