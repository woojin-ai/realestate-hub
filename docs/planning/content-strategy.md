# 콘텐츠 전략: 부동산 실거래가 대시보드 블로그/카테고리 기획

작성: 기획팀 / 대상: 디자인팀 → 개발팀
목적: 계산기 허브(`C:\claude code webside`)의 블로그 확장이 SEO 유입·체류시간·재방문에 효과가 있었던 패턴을 부동산 실거래가 대시보드에 맞게 재구성한다. 단, 부동산은 YMYL 성격이 강하므로 "투자 조언"으로 오해되지 않도록 구조 자체에 안전장치를 넣는다.

---

## 1. 콘텐츠 목표

- **SEO 유입**: 대시보드 자체는 "시/도+구 선택 → 조회" 형태라 검색엔진이 인덱싱할 만한 롱테일 키워드 페이지가 없다(클라이언트 상태 기반 SPA, `app/page.tsx` 참고). "OO구 아파트값" 같은 지역+시세 키워드는 검색량이 크므로, 정적으로 인덱싱되는 글이 이 유입을 대신 잡아야 한다.
- **체류시간·재방문**: 대시보드는 한 번 조회하고 나가는 "조회형" 세션이 되기 쉽다. 데이터 기반 글(추세 해설, 용어 설명)은 조회 후에도 읽을거리를 제공해 세션 길이를 늘리고, 정기 발행(예: 월간 시황 요약)은 재방문 유인이 된다.
- **부동산이라 계산기 허브와 다르게 주의할 점**:
  - 계산기 허브 블로그는 "계산 방법 설명"이라 사실관계가 명확했지만, 부동산 글은 "오를 것이다/사야 한다" 같은 표현이 한 줄만 들어가도 투자자문업 오인 소지가 생긴다. → **전망·추천·매수매도 권유 문구를 구조적으로 차단**해야 한다(4장 체크리스트).
  - 데이터 정확성 요구가 높다 — 계산기는 공식이 틀리면 바로 티가 나지만, 부동산 통계는 "그럴듯하게 틀려도" 독자가 못 알아챈다. 모든 수치 글은 반드시 대시보드가 실제로 계산한 값(국토부 원자료 기반)만 인용하고, 별도 추정치·외부 통계를 섞지 않는다.
  - `app/about/page.tsx`·`app/terms/page.tsx`에 이미 "참고용, 투자자문 아님, 원자료 확인 권장" 톤이 확립되어 있음 — 블로그도 이 톤을 그대로 승계한다(새로 만들지 않음).

## 2. 카테고리 구조 제안

계산기 허브의 `/salary /loan /date /life` 구조(계산기 성격별 분류)를 참고하되, 부동산은 "글의 목적"으로 나눈다. 4개 카테고리 제안.

| 카테고리 슬러그 | 이름 | 성격 | YMYL 위험도 |
|---|---|---|---|
| `/blog/insight` | 지역별 인사이트 | 특정 지역의 실거래가 흐름을 사실 기반으로 요약(전망 배제) | 중 — 표현 통제 필수 |
| `/blog/guide` | 용어·제도 설명 | 실거래가 신고 제도, 용어(전용면적, 계약갱신청구권 등), 사이트 데이터 읽는 법 | 낮음 |
| `/blog/howto` | 실거래가 활용법 | 대시보드 기능 자체를 활용하는 방법(예: "변동률 카드 읽는 법", "AI 추천 가중치 설정법") | 낮음 — 서비스 소개에 가까움 |
| `/blog/market` | 월간 시황 요약 | 전국/광역 단위 사실 기반 통계 요약(집계 수치만, 전망 없음) | 높음 — 가장 엄격한 검수 필요 |

- `insight`·`market`은 겹칠 수 있어 보이지만 스코프로 구분한다: `insight`는 "지역 1곳 딥다이브", `market`은 "여러 지역 비교/전국 개요".
- 계산기 허브처럼 카테고리를 4개로 유지해 목록 필터·색인 구조를 단순하게 가져간다(`lib/blog.ts`에 `BlogCategory` 유니온 타입으로 구현 — `planning/blog-spec.md` 패턴 재사용 가능).

## 3. 초기 글감 리스트 (카테고리별 3~5개, 제목 후보 포함)

우선순위는 **이 사이트가 이미 가진 데이터**(국토부 실거래가, 지하철역 거리, 학교 정보, AI 추천 로직)로 실제 작성 가능한 것부터.

### `/blog/insight` — 지역별 인사이트
1. "OO구 아파트 매매가, 1년 새 얼마나 올랐나 — 실거래가로 본 흐름" (특정 구 1곳, 최근 13개월 월별 평균가 차트 데이터 재구성 + 전월/1년 변동률 인용)
2. "역세권 아파트와 도보 10분 이상 아파트, 실거래가는 얼마나 차이날까" (지하철역 거리 데이터 활용, 특정 지역 단지 비교)
3. "신축과 구축, 같은 지역이라도 실거래가 차이는 이 정도" (준공연도별 그룹핑, 단지 상세 데이터 활용)
4. "매매가와 전세가, OO구에서는 격차가 어떻게 움직였나" (매매/전세 탭 데이터 비교)
5. "학군 인접 단지가 실제로 더 비쌀까 — 반경 1km 학교 데이터로 본 시세" (학교 정보 API 데이터 활용)

### `/blog/guide` — 용어·제도 설명
1. "실거래가 신고, 계약일로부터 며칠 안에 해야 할까 — 최근 달 데이터가 적어 보이는 이유"
2. "전용면적과 공급면적, 아파트 시세 비교할 때 헷갈리지 않는 법"
3. "국토교통부 실거래가 공개시스템, 이 사이트와 어떻게 다른가"
4. "매매·전세 실거래가 신고제, 알아두면 좋은 기본 용어 정리"

### `/blog/howto` — 실거래가 활용법
1. "변동률(전월·3개월·6개월·1년) 카드, 어떻게 읽어야 할까"
2. "AI 맞춤 추천 가중치(예산·역세권·신축) 제대로 설정하는 법"
3. "관심 지역 두 곳을 비교하려면 이렇게 조회하세요"
4. "단지 상세 페이지에서 놓치기 쉬운 정보 3가지 (세대수, 난방방식, 인근 학교)"

### `/blog/market` — 월간 시황 요약 (사실 기반, 전망·추천 배제)
1. "2026년 O월 전국 주요 지역 아파트 매매 실거래가 요약" (집계만, 해석 최소화)
2. "이번 달 거래량이 많았던 지역과 적었던 지역" (건수 기준 사실 나열)
3. "매매 대비 전세 비중, 지역별로 어떻게 다른가" (사실 기반 비교, 원인 추측 자제)

> `market` 카테고리는 정기 발행물이라 데이터 자동 집계 파이프라인이 필요할 수 있음(수작업 갱신은 지속 가능성 낮음) — 우선순위 로드맵에서 후순위로 미룬 이유이기도 함.

## 4. 리스크 체크리스트

**금지 표현 목록** (이 표현이 들어가면 투자자문으로 오인될 수 있어 게시 전 반드시 걸러낼 것):
- "지금 사야 한다", "매수 타이밍", "오를 것이다/떨어질 것이다" 등 미래 예측·단정
- "추천 단지", "유망 지역" 등 특정 매물·지역을 콕 집어 우위를 매기는 표현(AI 추천 기능 설명 글에서도 "점수화 결과"라고만 표현하고 "좋은 매물"이라 부르지 않음)
- "저평가/고평가" 같은 가치 판단 단어(사실은 "평균 대비 높음/낮음" 같은 중립 서술로 대체)
- "수익률", "투자가치", "시세차익" 등 투자 성과를 암시하는 단어

**데이터 출처·기준일 명시 원칙**:
- 모든 수치 인용 글은 본문 상단 또는 표 근처에 "국토교통부 실거래가 공개시스템 공공데이터 기반, 조회 기준일 YYYY-MM-DD" 표기 필수.
- 사이트가 계산하지 않은 외부 수치(언론 보도, 타 통계 등)는 인용 금지 — 대시보드가 실제로 보여주는 값만 쓴다(2장 참고, about 페이지 톤과 일치).
- "최근 달 데이터는 신고 지연으로 건수가 적을 수 있다"는 about/terms의 기존 면책을 데이터 인용 글마다 반복 고지.

**면책 문구 삽입 위치**:
- 계산기 허브의 `callout` 섹션 패턴(`type: "callout", variant: "warning"`)을 그대로 재사용 — 모든 글의 본문 최상단(도입부 직후) 1개 + `insight`/`market` 카테고리는 본문 중간 수치 인용 직후 추가 1개.
- 문구 예시(약관 제4조 톤 승계): "이 글에 포함된 실거래가 통계는 참고용 정보이며, 투자 자문이나 시세 보증을 목적으로 하지 않습니다. 매수·매도 등 의사결정 전 국토교통부 실거래가 공개시스템 원자료와 전문가 확인을 함께 거치시기 바랍니다."
- 글 하단에는 "이 글에 사용된 데이터를 대시보드에서 직접 조회해보기" CTA 링크 + 위 면책 문구 반복.

## 5. 우선순위 로드맵

- **MVP (1차)**: 카테고리 1개(`/blog/guide`)로 시작 — 위험도가 가장 낮고, 실거래가 대시보드 자체를 처음 쓰는 방문자에게 실질적 도움을 주며 서비스 신뢰도도 올린다. 글 2~3개(예: "신고 제도 설명", "면적 구분", "이 사이트와 국토부 시스템의 차이")로 오픈. 계산기 허브가 4개 카테고리를 한 번에 열지 않고 데이터 배열 방식(`lib/blog.ts` 패턴)으로 점진 확장했던 방식을 그대로 따른다.
- **2차**: `/blog/howto` 추가 — 대시보드 기능(변동률 카드, AI 추천) 활용법은 여전히 저위험이면서 서비스 이해도를 높여 이탈률을 낮춘다.
- **3차**: `/blog/insight` 추가 — 이 시점부터 표현 검수(4장 체크리스트)를 엄격히 적용. 특정 지역 1곳부터 시작해 템플릿을 검증한 뒤 확장.
- **4차 (보류 권장)**: `/blog/market` — 정기 발행 부담과 표현 위험이 가장 크므로, 앞 세 카테고리가 안정된 뒤 데이터 자동집계 방안까지 마련되면 착수. 이 순서는 다음 실행에서 뒤집힐 수 있으니 마스터 확인 후 확정.

## 6. 디자인팀에게 넘길 것

- **블로그 목록 페이지**: 카테고리 필터(4개 탭/드롭다운, 계산기 허브 카테고리 필터 UX 참고), 카드에 발행일·읽기시간·카테고리 배지 표시.
- **블로그 상세 페이지**: 브레드크럼(홈 / 블로그 / 카테고리), 본문 중간·하단 면책 콜아웃 박스(계산기 허브 `callout` 스타일 참고), 데이터 출처·기준일 표기 영역.
- **관련 콘텐츠 내부링크**: 계산기 허브는 "관련 계산기" CTA였지만 이 사이트는 대시보드가 지역 선택 기반 SPA라 고정 URL이 없다는 점에 유의 — 현재 `app/page.tsx`는 클라이언트 상태(`sido`, `gu`, `lawdCd`)로 지역을 다루고 지역별 정적 라우트가 없다. 따라서 "관련 지역 대시보드 링크"를 만들려면 다음 중 하나가 선행되어야 한다(개발팀 확인 필요, 기획팀이 임의 결정하지 않음):
  - (a) 쿼리스트링 기반 딥링크(`/?sido=서울특별시&gu=강남구`)를 대시보드가 초기 상태로 해석하도록 지원, 또는
  - (b) 지역별 정적/동적 라우트(`/region/[code]`) 신설.
  - 디자인팀은 우선 "관련 지역 바로가기" 버튼 자리만 UI에 확보해두고, 실제 링크 방식은 개발팀 확인 후 확정.
- **AI 추천/단지 상세 연결**: `howto` 카테고리 글에서 AI 추천 화면으로 보내는 CTA, `insight` 글에서 특정 단지 상세로 보내는 CTA도 위와 동일하게 "고정 URL이 있는 화면인지" 먼저 확인 후 링크 컴포넌트 설계.
- **면책 콜아웃 컴포넌트**: 계산기 허브의 `callout`(info/warning) 2종 스타일을 그대로 이식 추천 — 이 사이트는 warning 사용 빈도가 계산기 허브보다 높을 것(4장 참고).

## 7. [2026-07-18 추가] 뉴스 종합 시황 브리핑 — `/blog/market` 활성화 스펙

작성 배경: 사용자가 "아파트 매매/전세 관련 뉴스를 블로그 글 형식으로" 요청했다. 마스터가 확인한 방식은 **특정 기사 하나를 요약하는 게 아니라, 여러 뉴스의 사실(fact)만 재료로 삼아 완전히 새로운 문장으로 종합 작성**하는 것 — 저작권은 "표현"을 보호하지 사실 자체를 보호하지 않는다는 논리이며, 다음을 조건으로 승인됐다: ① 데이터 소스는 `NaverSearch-search_news`(응답이 제목+짧은 스니펫+원문링크뿐, 전문이 없어 구조적으로 베낄 수 없음), ② 기존 4장 리스크 체크리스트를 완화가 아니라 더 엄격히 적용, ③ 최소 3개 이상 서로 다른 뉴스 소스를 재료로 삼고 특정 기사의 표현/관점을 옮기지 않음, ④ 이 사이트 자체 실거래가 데이터(1차 데이터)로 뉴스의 사실을 교차검증.

### 7-0. 결정 — 신규 카테고리 신설 안 함, 기존 `/blog/market`을 이 방식으로 활성화

**`/blog/news` 같은 신규 카테고리를 만들지 않는다. 4장·5장에서 "4차, 보류 권장"으로 미뤄져 있던 기존 `/blog/market`을 이 뉴스 종합 + 자사 데이터 팩트체크 방식으로 활성화한다.** 근거:

1. **스키마가 이미 예약돼 있다.** `lib/blog.ts`의 `BlogCategory` 유니온은 이미 `"guide" | "insight" | "howto" | "market"` 4개로 확정돼 있고 `market`의 라벨(`"월간 시황"`)·배지색(`#6a1b9a`/`#f3e5f5`)도 이미 정의돼 있으나 실제 글은 0편이다(2026-07-18 기준 `docs/team-log/2026-07-18.md` 라운드25: guide 3편·howto 3편·insight 2편, market만 미개통). 신규 카테고리를 또 만들면 5번째 유니온 값·배지색·라벨을 새로 늘리는 셈이라, 애초에 "4개로 유지"하려던 §2의 설계 의도(계산기 허브 패턴 승계)에 어긋난다.
2. **두 개념은 사실상 겹친다.** "뉴스 종합 시황"과 기존 `market`("전국/광역 단위 사실 기반 통계 요약")은 둘 다 "부동산 시황"을 다뤄 같은 SEO 키워드·같은 독자 의도를 놓고 경쟁한다. 굳이 둘로 쪼개면 카테고리 탭에 비슷한 항목 2개가 나란히 떠서 독자만 혼란스럽다.
3. **폐기가 아니라 상위호환이다.** 원래 `market` 정의(§3의 글감: "집계만, 해석 최소화", "건수 기준 사실 나열")는 "이 사이트 데이터만 사용"이 원칙이었다. 이번 프로세스는 그 원칙(사이트 데이터가 최종 근거)을 그대로 유지하면서, 뉴스는 "이번 주 어떤 지역·이슈를 다룰지 고르는 계기"로만 보탠다(7-2). 원칙의 폐기가 아니라 검증 단계가 하나 더 붙는 강화다.
4. **`insight`와의 스코프 구분도 유지되고, 실무로 더 선명해졌다.** §2 원안은 insight=지역 1곳 딥다이브, market=여러 지역 비교/전국 개요로 나눴다. 실제 운영(`docs/team-log/2026-07-18.md` 라운드24·25)에서는 `insight`가 YMYL 리스크 때문에 "지역 실명·실제 수치 0, 해석 원칙만 다루는 상시(evergreen) 방법론 글"로 굳어졌다(`regional-price-difference-reading-principles`·`location-info-reading-principles` 둘 다 가상 예시만 사용, 실제 지역명 0). 이번에 활성화하는 `market`은 반대로 "이번 주 실제로 화제였던 지역·이슈 + 실제 수치"를 다루는, **시점에 묶인(time-bound)** 유일한 카테고리가 된다 — "상시 방법론 vs 이번 주 실사건"으로 축이 이미 분리돼 있어 실제로는 겹치지 않는다.

**결론**: 새 URL·새 유니온 값을 만들지 않는다. `lib/blog.ts`의 `blogPosts` 배열에 `category: "market"` 글을 append하는 기존 방식(guide/howto/insight 8편과 동일한 패턴) 그대로 쓴다 — IR(정보구조) 변경 없음.

**개발팀에 남기는 사소하지만 필요한 변경 1건**: `BLOG_CATEGORY_LABEL.market`의 표시 라벨을 `"월간 시황"`에서 `"시황 브리핑"`(가칭)으로 바꾸는 걸 권장한다 — 7-4에서 발행 주기를 "주 1회"로 제안하는데 라벨에 "월간"이 박혀 있으면 실제 주기와 문구가 어긋난다. `lib/blog.ts` 한 줄 변경이라 개발팀 확인 후 처리(기획팀은 코드를 건드리지 않음).

### 7-2. 작성 프로세스 (주 1회 반복)

**1단계 — 광역 스캔(이번 주 화제 지역·이슈 파악)**
- `NaverSearch-search_news`를 `sort: "date"`(최신순), `display: 20~30` 정도로 호출:
  - `"아파트 매매"`, `"아파트 전세"` (매주 고정 기본 2종)
  - 시점에 따라 `"부동산 규제"`, `"아파트값"`, `"전세 매물"` 등 보조 키워드 1~2개 추가(합쳐서 2~4개 조합).
- 결과 중 **발행일이 최근 7일 이내인 것만** 채택(주간 브리핑이므로 오래된 기사 배제).
- 제목·스니펫에 반복 등장하는 지역명·이슈를 추린다 — 이번 주 소재 후보.

> **[2026-07-19 추가 · 마스터 승인] 시계열 소재 예외 — 소스 창 최대 3주 확대**
>
> **주간 단위 지표의 3주 이상 시계열을 다루는 편에 한해, 가장 최근 데이터 포인트가 7일 창 안에 있으면 소스 창을 최대 3주까지 확대할 수 있다. 확대 시 그 사유를 기획안에 기록한다.**
>
> 배경(라운드 29 "세종 전세" 편): 소재의 핵심 사실 자체가 주간 변동률 3주 추이(0.10% → 0.23% → 0.04%)였는데, 주간 지표는 매체가 매주 한 번씩 보도하므로 비교 구간 기사는 구조적으로 7일 창 밖에 있을 수밖에 없다. 7일 창을 문면대로 적용하면 3소스 요건(7-3-7)을 못 채워 결번인데, 정작 최신 데이터 포인트를 다룬 기사는 창 안에 있어 신선도라는 규칙의 **취지**는 충족되는 모순이 생겼다. 규칙의 목적(오래된 화제 배제)과 수단(발행일 7일)이 어긋난 사례라 예외를 명문화한다.
>
> 남용 방지 조건 — 아래를 모두 만족할 때만 적용한다:
> 1. **가장 최근 데이터 포인트를 다룬 기사가 7일 창 안에 있을 것**(신선도의 핵심은 최신 관측치이지 비교 구간이 아니다).
> 2. **시계열 자체가 소재일 것** — 단순히 소스가 부족해서 창을 넓히는 용도로 쓰지 않는다. "이번 주 단독 사건"을 다루는 편은 기존 7일 창을 그대로 적용한다.
> 3. **확대는 최대 3주까지.** 그 이상 과거 기사는 배경 이해용으로만 쓰고 참고 기사 목록(5단계)에 넣지 않는다.
> 4. **글 도입부에서 "최근 N주 추이"임을 독자에게 명시**할 것 — 독자가 이번 주 단독 사건으로 오인하지 않게 한다.
> 5. 확대해도 **3소스 최소 요건(7-3-7)은 그대로** 적용된다 — 요건을 낮추는 예외가 아니라, 창을 넓혀 요건을 정상적으로 충족시키는 예외다.

**2단계 — 정밀 스캔(후보 지역·이슈로 재검색)**
- 1단계 후보 지역명·이슈로 재검색(예: `"{지역명} 아파트"`, `"{지역명} 전세"`), `sort:"date"`와 `sort:"sim"` 둘 다 시도.
- 모은 스니펫에서 **최소 3개 이상 서로 다른 언론사**가 공통으로 언급하는 사실만 남긴다.
  - "서로 다른 언론사" 판정: 실제 취재·작성 주체가 다른 3곳이어야 한다 — 통신사(연합뉴스 등) 기사를 여러 매체가 그대로 전재한 경우는 1개 소스로 간주(7-3-2).
  - 채택 예: "OO구 거래량이 전월 대비 늘었다", "정부가 OO 규제를 발표했다" 같은 **검증 가능한 사실 진술**만. "오를 것이다" 류 전망성 문장은 이 단계에서 재료 후보에서 제외.

**3단계 — 사이트 자체 데이터로 교차검증(가능하면 필수)**
- 뉴스가 언급한 지역명을 이 사이트의 지역명→법정동코드 매핑 자산(`lib/molit-api.ts`의 지역 코드 테이블 등 — 정확한 식별자는 개발팀 확인)으로 `lawd_cd`를 찾는다.
- 기존 라이브 엔드포인트 `/api/data?lawd_cd={code}&months=13&building_type=아파트`(`app/api/data/route.ts`)로 최근월 거래건수·평균가·전월 대비 변동률을 확인한다.
- **일치**: "OO매체 보도대로 이 사이트 데이터로도 거래량 증가가 확인된다"처럼 자사 데이터를 근거로 명시.
- **불일치·확인 불가**: 무리하게 끼워맞추지 않는다. "언론 보도와 달리 이 사이트 데이터 기준으로는 ~"로 정직하게 병기하거나, 그 사실을 아예 소재에서 제외한다. **뉴스 사실이 자사 데이터로 확인 안 되면 그 문장은 쓰지 않는 게 기본값**(§4 "외부 수치 인용 금지" 원칙과 이번 프로세스를 잇는 핵심 다리).
- 참고: `/ranking`(전국 시군구 랭킹 보드, `docs/planning/next-features-2026-07-17.md`, 아직 마스터 승인 대기)이 배포되면 "이번 주 상승·하락 상위 지역"을 한 번에 뽑아 1단계 소재 선정과 3단계 교차검증을 동시에 훨씬 빠르게 할 수 있다 — 향후 시너지로 기록.

> **[2026-07-19 추가] 불일치는 결함이 아니라 이 카테고리의 최대 자산이다.**
>
> 라운드 29 "세종 전세" 편에서 여러 매체가 공통으로 전한 "전세 실종·매매 쏠림" 서사가 자사 실거래 데이터로는 **반박**됐다(완결된 최근 12개월 내내 전세 거래건수 > 매매 거래건수). 이때 소재를 버리거나 뉴스 쪽에 맞추는 대신, 어긋남 자체를 본문의 핵심으로 삼고 "왜 어긋나는지"는 단정하지 않는 방식으로 처리했다. 이 사이트가 1차 데이터를 가진 이상 뉴스 재탕과 차별화되는 지점이 바로 여기이므로, **불일치가 나오면 소재 폐기를 먼저 고려하지 말고 "불일치 자체를 다루는 편"으로 재구성할 수 있는지 먼저 검토**한다. 단 다음을 지킨다:
> - 어느 쪽이 틀렸다고 **판정하지 않는다**. 집계 대상·모집단·지표 정의 차이를 병기하고 판단은 독자에게 남긴다.
> - 지표 정의가 다른 값(예: 한국부동산원 주간 지수 vs 자사 월평균 실거래가)은 **같은 표에 나란히 놓지 않는다** — 독자가 직접 비교 가능한 값으로 오인한다.
> - 비교 자체가 부적절한 경우(예: 갱신이 섞인 전세 건수 vs 그런 구분이 없는 매매 건수) 그 **부적절함을 본문에 명시**한다.

**4단계 — 집필(완전히 새로운 문장)**
- 3단계까지 걸러진 사실만 재료로 처음부터 새 문장으로 쓴다. 기사 문장 재구성·짜깁기 금지.
- 직접인용이 꼭 필요하면 **15단어 이내** + 매체명 명시(예: "OO경제는 '...'라고 전했다"). "한 매체에 따르면"처럼 흐리지 않는다.
- 매체별 해석이 갈리면 단정하지 않고 "다양한 시각이 있다"는 균형 서술(7-3).

**5단계 — 참고 기사 목록화**
- 본문 하단에 **실제로 재료로 쓴 기사만**(수집했지만 안 쓴 건 제외) 매체명+제목+원문 링크로 나열 — 출처 투명성 + 저작권 예우.

**6단계 — 면책 콜아웃 + 데이터 출처 표기**
- 기존 원칙 그대로: 도입부 직후 warning 콜아웃 1개 + 3단계 수치 인용 직후 콜아웃/출처캡션 추가 1개(`docs/design/blog.md` §2-3·§5).
- 데이터 출처 캡션은 기존 형식("국토교통부 실거래가 공개시스템 공공데이터 기반, 조회 기준일 YYYY-MM-DD") 그대로 쓰고, **뉴스 자체는 이 캡션 대상이 아니라 5단계 "참고 기사" 목록으로 별도 처리** — 두 출처(공공데이터 vs 뉴스)를 표기 위치·형식으로 명확히 분리해 독자가 "이 숫자는 국토부 데이터, 이 맥락은 언론 보도"를 헷갈리지 않게 한다.

### 7-3. 리스크 체크리스트 확장 — 뉴스 종합 콘텐츠 전용 추가 규칙

기존 4장 금지 표현 목록(오를 것이다/사야 한다/추천/저평가/수익률 등)은 이 카테고리에도 **완화 없이 그대로, 오히려 더 엄격히** 적용한다. 아래는 외부 뉴스가 재료로 섞이는 이 카테고리 전용 추가 규칙이다.

1. **출처 흐리기 금지**: "언론 보도에 따르면", "업계에 따르면", "한 매체는" 같은 뭉뚱그린 표현 금지 — 반드시 구체적 매체명을 명시한다(저작권법상 정당 인용 요건이자, 독자가 원문을 직접 검증할 수 있게 하는 신뢰도 장치).
2. **동일 소스 재전재 구분**: 통신사 기사를 그대로 받아쓴 여러 매체는 "서로 다른 소스 3곳"으로 세지 않는다 — 실제 취재·작성 주체가 다른 3곳이어야 최소 소스 요건 충족.
3. **특정 매체 논조 대변 금지**: 특정 매체의 분석·전망을 이 블로그의 결론처럼 서술하지 않는다. 사실(발표·수치·통계)과 그 매체의 해석·전망을 문장 단위로 분리하고, 해석·전망을 넣어야 하면 "OO매체는 ~라고 분석했다"처럼 귀속을 명확히 한다.
4. **시각이 갈릴 때 균형 서술**: 매체마다 해석이 다르면(예: "거래 회복" vs "여전히 침체") 단정하지 말고 "다양한 시각이 있다"/"매체별로 진단이 엇갈린다"고 병기한다.
5. **전망성 인용 재료 배제**: 뉴스 스니펫에 "오를 것", "추천" 같은 표현이 있어도 그대로 재인용하지 않는다 — 그 매체의 전망 자체를 소재에서 빼거나, 굳이 언급해야 하면 귀속된 인용으로만 다루고 이 사이트 본문의 주장으로 만들지 않는다.
6. **특정 단지·매물 실명 배제**: 뉴스가 특정 아파트 단지를 짚었더라도 이 카테고리는 **지역(시군구) 단위까지만** 다룬다(기존 §4 "추천 단지" 금지, `insight` 카테고리 기존 관행과 정합).
7. **최소 3소스 미달 시 미발행**: 교차확인되는 사실이 3개 미만이면 그 사실은 이번 주 소재에서 제외한다 — 무리하게 1~2개 기사로 채우지 않는다.
8. **자사 데이터 불일치 시 사이트 데이터 우선**(7-2 3단계 원칙 반복 — 가장 중요한 원칙이라 체크리스트에도 명시): 뉴스 사실이 사이트 데이터로 확인 안 되면 그 문장은 쓰지 않거나 불일치를 정직하게 밝힌다.
9. **참고 기사 링크 필수**: 출처 목록 없는 뉴스 종합 글은 발행하지 않는다.
10. **[2026-07-19 추가] 출처 신뢰도가 흔들리는 수치는 전면 배제한다**: 아래 중 하나라도 해당하면 그 기사의 수치를 본문에 옮기지 말고, 해당 기사는 "그런 보도가 있었다"는 정성적 사실 확인용으로만 쓴다.
    - 확인 경로마다 수치가 다르게 나온다(라운드 29 뉴시스 기사 — 검색 스니펫과 본문 요약에서 전세/매매 수치가 서로 뒤바뀌어 나왔다).
    - **기사가 통계 출처 기관명을 틀리게 적고 있다.** 위 뉴시스 기사는 마스터가 원문을 직접 열어 수치 방향을 확정했음에도(매매 580건·전세 150건, 매매지수 99.6·전세지수 101.8, 2026년 5월 말 기준) **최종적으로 전면 배제**했는데, 결정적 사유가 이것이었다 — 원문이 통계 출처를 "서울부동산정보원"으로 적고 있으나 주간 아파트가격 동향을 내는 기관은 한국부동산원이다. 기관명이 부정확한 기사의 수치를 옮기면 이 사이트가 그 오류를 재생산하게 된다(§7-3-1 취지).
    - 자사 데이터와 방향이 반대인데 어느 쪽이 맞는지 판정할 근거가 없다(이 경우 7-2 3단계 "불일치는 최대 자산"대로 **불일치 자체를 다루되 상대 수치는 인용하지 않는다**).
    > 즉 "원문 확인 = 인용 가능"이 아니다. 확인은 최소 조건일 뿐이고, 확인 결과 신뢰도 문제가 드러나면 배제가 맞다.
11. **[2026-07-19 추가] 부분월(월중)을 기준점으로 삼은 변동률은 본문에 쓰지 않는다**: 조회 시점이 월 중이면 그 달은 신고 지연까지 겹쳐 과소집계 상태이고, 화면의 변동률 카드 4종은 **전부** 가장 최근 달을 기준점(종점)으로 계산한다. 따라서 "전월 대비"만의 문제가 아니다.
    - **라운드 29 실측**: 같은 날 반나절 간격 두 조회에서 세종 전세 1년 변동률이 +18.04% → +18.73%, 매매 6개월이 -15.31% → -16.52%로 움직였다(월중 달의 집계 상태 차이). 본문에 하드코딩하면 독자가 보는 화면값과 어긋난다.
    - → **월중에 발행하는 편은 변동률 카드 수치를 인용하지 않는 것을 기본값**으로 하고, 대신 완결된 달의 월평균·거래건수 시계열로 서술한다. "변동률 카드는 진행 중인 달을 기준점으로 삼으므로 이 글에서는 인용하지 않는다"는 설명 자체를 본문에 넣으면 독자에게 유용한 해석 원칙이 된다.
    - 부득이 인용해야 하면 6개월·1년처럼 부분월 영향이 희석되는 구간만 쓰고, **해당 월이 월중임을 본문에 명시**한다.

### 7-4. 발행 주기 / 운영 부담 판단

기존 §5가 `market`을 "정기 발행 부담과 표현 위험이 가장 크다"며 4차·보류로 미뤘던 이유를 재검토한다.

- **이미 있는 자동화 인프라**: `realestate-team-standup` 스케줄(`0,30 1-7 * * *`, 매일 01:00~07:30 사이 30분 간격 14회)이 상시 가동 중이고, 최근 라운드(19~25)에서 이미 `howto` 3편·`insight` 2편을 이 자동화 세션이 "IR 무변경 안전 트랙"으로 실제 생산해왔다(`docs/team-log/2026-07-18.md`). 정기적으로 글을 만들어내는 실행력 자체는 이미 검증됐다.
- **그럼에도 `market`은 기존 8편과 질적으로 다른 부담이 있다**:
  1. 기존 글(guide/howto/insight)은 "이 사이트 코드·기능을 Read해서 설명"하는 폐쇄형 소재라 하루 중 아무 라운드에서나 착수해도 품질이 그대로였다. `market`은 매번 그 시점의 최신 뉴스를 검색해야 하는 개방형 소재라, 아무 라운드에나 반복 시도하면 같은 주 안에서 검색 결과가 겹치거나 신선도가 무의미해진다.
  2. 사실 교차확인(최소 3소스)·자사 데이터 대조(3단계)는 기존 append-only 작업보다 편집 판단이 훨씬 많이 든다 — QA 게이트 항목도 늘어난다(7-3 전부 + 참고 기사 링크 유효성 확인).
  3. 매주 반드시 "교차확인 가능한 화제"가 있으리라는 보장이 없다.
- **제안: 주 1회**, 주간 스케줄 중 **특정 요일 첫 라운드 1회만** `market` 작성을 시도(예: 매주 월요일 01:00 라운드 — 그 전 한 주치 뉴스를 다루는 "주간 브리핑" 프레이밍과 자연스럽게 맞음). 같은 주 나머지 라운드는 기존 guide/howto/insight append·정비 작업을 계속한다.
  - **결번 허용**: 교차확인되는 사실이 3소스 미만이면 그 주는 결번(skip)하고 다음 주로 넘긴다(7-3-7과 직결).
  - 이 주기는 제안이며, 실제 운영해보고 마스터가 조정 가능(월 2회로 줄이거나 반대로 유지 등 재량).
- **자동화 세션에 반영할 것**(마스터/자동화 프롬프트 판단 필요 — 기획팀이 직접 cron 스케줄을 바꾸지 않음): "이번 라운드가 이번 주 첫 월요일 새벽 라운드인가"를 판별해 `market` 시도 여부를 정하는 로직은 cron 표현식이 아니라 자동화 세션의 판단 프롬프트에 넣는 편이 낫다.

### 7-5. 디자인팀에게 넘길 것

기존 블로그 상세 페이지 틀(`docs/design/blog.md`)을 그대로 재사용한다. 신규 요구사항은 "참고 기사" 출처 링크 블록 하나뿐 — 레이아웃 설계는 디자인팀 몫이므로 요구사항만 남긴다.

- **Callout(warning/info)**: 신규 없음. 기존 컴포넌트·색상 그대로(`docs/design/blog.md` §3).
- **카테고리 배지**: 기존 `market` 배지(`#6a1b9a` / `#f3e5f5`) 그대로. 단, 라벨 텍스트가 "월간 시황" → "시황 브리핑"(가칭, 7-0)으로 바뀔 수 있음을 미리 알아둘 것 — 배지 색·모양 변경은 없고 텍스트만 바뀐다.
- **신규: "참고 기사" 출처 목록 블록** — 요구사항만 명시, 배치·스타일은 디자인팀 판단:
  - 위치: 본문 최하단, 기존 하단 CTA 박스(`docs/design/blog.md` 2-4)와 "목록으로 돌아가기"(2-5) 사이 어딘가 — 정확한 순서(CTA 앞/뒤)는 디자인팀이 결정.
  - 내용: 소제목("참고 기사" 등) + 목록. 각 항목은 **매체명 + 기사 제목 + 원문 링크** 3요소 필수.
  - 외부 링크이므로 새 창 열림(`target="_blank" rel="noopener noreferrer"`) 처리 필요 — 이 사이트에 이미 있는 외부 링크 접근성 패턴(크로스 사이트 내비 위젯의 "새 탭에서 열림" 문구, `docs/team-log/2026-07-18.md` 라운드19)을 재사용 권장.
  - 콜아웃과 마찬가지로 이 블록도 단순 목록만 담고 표·차트 등 복잡한 구조는 넣지 않는다(`docs/design/blog.md` §3-3 공통 규칙과 동일 원칙).
- **개발팀 확인 필요(디자인 범위 밖, 참고용으로 남김)**: 현재 `BlogSection` 유니온(`lib/blog.ts`)에는 "링크 목록"을 표현할 타입이 없다(`list`는 `items: string[]`라 앵커 태그를 못 담음). "참고 기사"를 실제 클릭 가능한 링크로 만들려면 `{ type: "sourceLinks"; items: { outlet: string; title: string; url: string }[] }` 같은 신규 섹션 타입 추가가 필요해 보인다 — 기획팀이 결정할 사항이 아니라 개발팀이 스키마 변경 여부를 판단할 몫이므로 여기서는 "필요성 확인"까지만 남긴다(임시 대안: `list`에 "매체명 — 제목 (URL)" 평문 문자열을 넣는 방법도 있으나 클릭 가능한 링크는 아니다).

### 7-6. 마스터 확인 필요

- `BLOG_CATEGORY_LABEL.market` 라벨을 "월간 시황"→"시황 브리핑"(가칭)으로 바꿀지, 다른 문구를 원하는지.
- 발행 주기 "주 1회, 매주 월요일 첫 라운드" 제안 승인 여부(7-4) — 자동화 프롬프트에 반영하려면 마스터의 스케줄 판단 로직 반영이 필요.
- `BlogSection`에 "참고 기사" 링크 목록용 신규 타입을 추가할지(7-5 개발팀 확인 필요 항목) — 승인되면 다음 라운드 개발팀에 전달.
- 이 카테고리는 외부 뉴스가 재료로 들어가는 유일한 카테고리라, 첫 1~2편은 자동화 세션 단독 판단이 아니라 **마스터가 직접 최종 검수(참고 기사 링크 실제 클릭 유효성 확인 포함)한 뒤 게시**하는 걸 권장 — qa-team 게이트에 더해 마스터 확인 1단계를 임시로 추가하는 안. 안정화되면(예: 4~5편 무사고) 기존 8편과 동일하게 자동화 세션 자율 진행으로 전환 검토.
