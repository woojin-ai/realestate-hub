## 1. 진입점 / 배치 / 접힘·펼침 흐름

### 1-1. 렌더 위치
`app/page.tsx`에서 **`<DealsTable>` 바로 아래**에 새 `<AiRecommendSection>`을 배치한다. 조회 결과 블록(`status === "ready" && data && !isEmptyResult`) 안에만 렌더한다.

```
app/page.tsx (152~161줄 블록 내부)
  <DealTypeTabs .../>
  <SummaryCards .../>
  <MonthlyChart .../>
  <DealsTable rows={rows} dealType={dealType} />
  <AiRecommendSection            ← 신규
     lawdCd={lawdCd}
     gu={gu}
     dealType={dealType}
     rows={rows}                 ← 후보 풀(이미 메모리에 있는 AptStat[])
  />
```

### 1-2. 노출 조건(중요)
- **조회 전(지역 미선택 or 아직 조회 안 함, `status !== "ready"`)**: 섹션 **자체를 렌더하지 않음**. 부모의 `status === "ready" && data && !isEmptyResult` 가드 안에 두므로 자동 충족.
- `isEmptyResult`(데이터 없음)일 때도 미노출(부모 분기가 `<>...</>` 자체를 안 그림).
- 건물유형이 아파트가 아닐 때: 현재 추천 로직은 아파트 단지 통계(`AptStat`) 전제. **빌라·단독 탭에서는 섹션을 렌더하지 않는다**(`buildingType === "아파트"`일 때만). page.tsx에서 `buildingType` 전달하거나, rows가 비면 내부에서 자체 미노출. → 개발 시 `buildingType !== "아파트"`면 `return null`.

### 1-3. 접힘/펼침 상태머신
초기 = **접힘(collapsed)**. 불필요한 API/연산을 조회 즉시 돌리지 않는다(기획 §4).

```
[collapsed]  기본. "AI 맞춤 추천 보기" 버튼 배너만 노출
    │  버튼 클릭
    ▼
[form]       입력폼 펼침(예산/연도/가중치). 아직 실행 전. "추천 받기" CTA
    │  "추천 받기" 클릭
    ▼
[running]    부분결과 스켈레톤/진행 배지 (§3)
    │  결과 도착(부분→전체)
    ▼
[result]     카드 그리드 + 입력폼 상단 유지(슬라이더 즉시 재정렬 §3-B)
    │  "접기" / 지역 재조회 시
    ▼
[collapsed]  (지역·거래탭 변경 시 result 폐기 → §6)
```

- 접힘 배너: 대시보드 카드 톤과 통일 — `bg-white rounded-xl shadow-sm p-4 md:p-5 mb-5`. 좌측 아이콘 `✨`/`🤖`, 제목 "AI 맞춤 추천", 보조 카피 "예산·역세권·신축 선호도로 이 지역 단지를 점수화해 추천해요", 우측/하단 CTA 버튼.
- CTA 버튼 스타일: 기존 조회 버튼과 동일 계열 `rounded-lg bg-brand px-7 py-2.5 text-sm text-white hover:bg-[#303f9f] min-h-[44px]`.
- 펼침/접힘은 애니메이션 없어도 무방(스크롤 점프 방지 위해 펼칠 때 섹션 상단으로 스크롤은 하지 않음). `aria-expanded`를 배너 버튼에 부여.

## 2. 입력 폼

기획 §1 MVP 입력: **예산 + 예산유형(이하/이상) + 신축기준연도 + 가중치 3종**. 거래유형은 대시보드 상단 `DealTypeTabs`와 연동(폼에 별도 컨트롤 두지 않음, 현재 `dealType` prop 사용).

### 2-1. 필드 스펙

| 필드 | 컨트롤 | 기본값 | 범위/옵션 | 비고 |
|---|---|---|---|---|
| 예산 | 숫자 입력 + 단위 표기(억/만원) | 지역 평균가 근처로 프리필 권장(예: `summary.current_avg` 반올림) or 비움 | 0 초과. 억 단위 스텝 UX 권장 | 원본 `rec-budget`. 입력 placeholder "예: 8억" |
| 예산유형 | 셀렉트/토글(이하·이상) | **이하(max)** | `max`=이하 / `min`=이상 | 원본 `budget_type`. "이하"=예산 넘는 단지 감점/제외 |
| 신축기준연도 | 숫자 입력 or 셀렉트(연도) | **2015**(현재연도−10 권장) | 1980 ~ 현재연도 | 원본 `year`. "이 연도 이후 준공을 신축으로 우대" 툴팁 |
| 거래유형 | (폼 밖) 대시보드 탭 연동 | 현재 탭 | 매매/전세 | 폼에 표기만: "현재 '매매' 기준으로 추천" 회색 라벨. [확인 필요: 전세 시 가격점수 의미 §0] |
| 가중치·가격 | 슬라이더 | **45** | 0~100 | 색상 `up`(#e53935) 계열 힌트 |
| 가중치·역세권 | 슬라이더 | **35** | 0~100 | 색상 `down`(#1e88e5) 계열 힌트 |
| 가중치·신축 | 슬라이더 | **20** | 0~100 | 색상 초록(#43a047) 계열 힌트 |
| ~~가중치·평지~~ | **없음(MVP 제외)** | — | — | §2-3 참고 |
| ~~세대수 필터~~ | **없음(MVP 제외)** | — | — | 기획 §1. UI에 노출 안 함 |

- **가중치 합 처리**: 원본은 슬라이더 합 100% 가정. MVP는 3종이므로 **입력값을 합=100으로 강제하지 않고**, 표시 옆에 현재 정규화 결과를 안내: "가격 45 · 역세권 35 · 신축 20 → 합 100%". 세 값 합이 100이 아니어도 점수 계산 시 **각 값/합으로 정규화**(dev). 화면엔 각 슬라이더 옆 현재 정수값을 노출하고, 하단에 작은 회색 텍스트로 "가중치는 상대 비율로 자동 환산됩니다".
- 각 슬라이더는 라벨(아이콘+이름) + 현재값 + `<input type="range">`. 접근성: `<label>` 연결, `aria-valuetext`에 "가격 선호 45" 형태.

### 2-2. 모바일/데스크톱 레이아웃
- **데스크톱(≥768px)**: 예산·예산유형·신축연도를 한 줄 그리드(`grid grid-cols-[1fr_auto_1fr] gap-3` 수준), 그 아래 가중치 슬라이더 3개를 가로 3열(`md:grid-cols-3 gap-4`), 우하단에 "추천 받기" CTA.
- **모바일(<768px)**: 전부 1열 세로 스택. 각 필드 `min-h-[44px]`. 슬라이더는 풀폭. CTA는 풀폭 버튼(`w-full`).
- 폼 컨테이너: 카드 안 서브영역 `border-t border-[#e8eaf6] pt-4 mt-3` 정도로 헤더와 구분.

### 2-3. 평지 가중치 처리 (MVP 제외 방식 확정)
기획 §1·§6-6 권고 = **완전 제거**. 화면 결정:
- 가중치 슬라이더에서 **평지 항목을 렌더하지 않는다**(4번째 슬라이더 없음).
- 카드에서도 평지 점수 막대 제거(§4).
- "준비중" 뱃지/중립값 표기 **하지 않음**(기획: 오해 소지). 대신 섹션 헤더 보조 카피나 폼 하단에 한 줄 안내를 **선택적으로** 둘 수 있음: "※ 지형(평지) 분석은 준비 중입니다" — 단, 오해를 줄이려면 **아예 언급 안 하는 쪽을 1순위 권고**. 최종은 마스터 확인 불필요(기획 권고 범위 내), 개발은 "평지 UI 없음"으로 구현.

## 3. 로딩 / 점진적 표시 UX

두 개의 서로 다른 인터랙션을 명확히 구분한다: **(A) 최초 추천 실행(외부호출 가능)** vs **(B) 가중치 재조정(외부호출 0)**.

### 3-A. 최초 실행 — 캐시 상태에 따른 3가지 경로

기획 §2 추천안: 캐시 히트는 즉시, 미스는 이번 라운드 최대 `K`개만 채우고 점진 완성.

| 경로 | 상황 | 화면 표현 |
|---|---|---|
| ① 웜(전부 캐시 히트) | `apt_geo` 다 있음, 외부호출 0 | 진행바 없이 **즉시 카드 그리드**. 실행 버튼 클릭 후 200~수백 ms 내 결과 → 스피너 대신 버튼에 잠깐 로딩 상태만 |
| ② 부분 캐시 | 일부만 히트 | 히트분 카드 **먼저 렌더** + 상단 **진행 배지**: "이 지역 데이터를 채우는 중 · N/M" + 미채움 슬롯 **스켈레톤 카드** |
| ③ 콜드(전부 미스) | 첫 조회 지역 | **스켈레톤 카드 K개** + 진행 배지 "지역 데이터 준비 중…". 1라운드 채움(최대 K) 후 부분결과 표시, 남으면 "더 채우기" or 자동 다음 배치 |

- **스켈레톤 카드**: 실제 카드와 동일 골격(메달 자리 원, 단지명 바, 점수 자리 바, 막대 3줄 회색). `animate-pulse bg-gray-100`. 실제 카드 높이와 맞춰 레이아웃 점프 방지.
- **진행 배지**: 대시보드 로딩 배너 톤 재사용 — `bg-[#f5f7ff] border border-[#dde1f5] rounded-lg px-4 py-3 text-sm text-brand`, 좌측 스피너(`animate-spin` 원), 텍스트 "이 지역 추천 데이터를 준비하고 있어요 · 3/12". 진행 수치는 SSE든 폴링이든 dev 구현에 위임 — 화면은 `{done}/{total}` 숫자와 문구만 요구.
- **부분결과 우선 노출 원칙**(기획 §2-3): "채워진 것부터 즉시 카드로", 나머지는 스켈레톤/다음 배치. 콜드에서 1~2분 흰 화면 금지.
- **콜드 채움 상한/배치 크기 `K`**: `[확인 필요 — Vercel max duration]`. 화면은 `K` 값에 하드 의존하지 않음(스켈레톤 개수 = "이번 라운드 대상 수"로 dev가 전달).
- **채움 미완 안내**: 예산·필터로 후보가 K를 초과해 이번 라운드에 다 못 채운 경우, 그리드 하단에 회색 안내 + (선택)"남은 단지 더 불러오기" 버튼. 자동 다음 배치로 갈지 수동 버튼일지는 dev 성능 실측 후 결정 — **화면은 두 경우 모두 수용**(버튼 있으면 렌더, 없으면 미표시).

### 3-B. 가중치 재조정 — 외부호출 없이 즉시 재정렬 (핵심 UX)

기획 §2-4: 슬라이더 재조정은 캐시된 geo만으로 **클라이언트 즉시 재계산**. 카카오 재호출 없음.

화면에서 이 "즉시성/무비용"을 어떻게 알리나:
- **로딩/진행바를 띄우지 않는다.** 슬라이더를 움직이면 카드 순서·점수·막대가 **부드럽게 즉시 갱신**된다. (진행 배지 절대 재등장 금지 — 그러면 "또 오래 걸리나" 오해)
- 재정렬 시 카드에 **위치 이동 트랜지션**(옵션, `transition` + reorder) — 순위가 바뀌는 걸 시각적으로 보여주면 "내 선호가 반영됐다"가 직관적. 과하면 생략 가능(성능 우선).
- 폼 하단에 상시 마이크로카피: **"슬라이더를 움직이면 추가 조회 없이 바로 다시 정렬돼요"** (`text-xs text-gray-400`). 사용자가 "재조정=비용/대기"로 오인하지 않게 하는 핵심 문구.
- 점수 숫자/막대 폭은 재계산 결과로 갱신. 종합점수는 3요소 가중합 → 슬라이더 따라 실시간 변동.
- 최초 결과가 **부분결과(일부 스켈레톤)** 인 상태에서 슬라이더를 움직이면: **이미 채워진 카드만 재정렬**하고 스켈레톤은 그대로(채움은 별개 트랙). 이 동작을 진행 배지 문구로 구분("정렬은 즉시, 데이터 채움은 진행 중").

> 개발 참고: 재정렬은 API 재호출 없이 클라이언트 상태(캐시된 각 단지의 price/subway/new 원점수)만으로 `score = Σ(정규화가중치 × 원점수)` 재계산 → 정렬. 즉 **API 응답은 "가중치 적용 전 원점수(price_score·subway_score·newbuild_score)"를 반드시 포함**해야 슬라이더 무호출 재정렬이 가능(§7 데이터계약 필수 필드).

### 4-B. 세부 결정

- **지하철 배지 미확인 처리**: 원본은 `subway_dist < 9999` 조건. MVP도 동일 — 거리 없으면 "도보 미확인". 캐시 미스로 아직 안 채워진 단지는 애초에 스켈레톤이므로 카드로 안 뜸(채워진 것만 카드).
- **카드 내 평수별 시세**: 원본은 카드 안에 area_stats 미니표 + "예산내" 초록 태그를 넣었다. 신 프로젝트는 **동일 정보를 `AptDetailModal`(평수별 상세표)이 이미 제공**하므로, **카드 안에는 중복 미니표를 넣지 않고** 대신 요약 1줄만 둔다: 예 "예산 이하 평형 N개" 또는 "대표 59평 8.5억". 근거: (1) 카드 높이 억제로 그리드 스캔성↑, (2) 상세는 클릭→모달로 위임해 정보 계층 명확화, (3) 모달과 미니표가 같은 데이터를 두 번 그리는 유지비 제거.
  - 단, 원본의 "예산내 강조"는 사용자 가치가 크므로 **모달 쪽에 예산 하이라이트를 이관**하는 것을 §4-C에서 다룸.
- **점수 막대 컴포넌트**: 라벨 + 우측 점수 + 하단 막대(`bg-[#eef] rounded h-1.5` wrap, `h-1.5 rounded` fg, `width: {score}%`). 접근성 `role="img" aria-label="가격 점수 72점"`.
- 카드 컨테이너: `bg-white rounded-xl border border-[#e8eaf6] p-4 shadow-sm hover:shadow-md transition cursor-pointer`. 1~3위는 살짝 강조(원본 `rank-1..3`): 좌측 보더 컬러 or 미묘한 배경(예 1위 `ring-1 ring-amber-200`). 과한 금테는 지양.

### 4-C. 카드 클릭 → `AptDetailModal` 재사용 (핵심 결정)

**결론: 재사용한다. 단, 조건과 데이터 정합성 처리를 명시.**

- `AptDetailModal`의 props는 `{ apt: AptStat; onClose }`(`docs/design/apt-detail-modal.md` §7). 추천 카드의 원본 데이터가 **`rows`(부모가 넘긴 `AptStat[]`)에서 온 것**이므로, 추천 API가 반환한 확장 객체가 아니라 **원본 `AptStat` 참조를 그대로 모달에 넘기면 된다**. → 모달은 수정 없이 재사용 가능.
  - 매핑: 추천 결과 항목은 `name`+`dong`을 키로 부모 `rows`에서 원본 `AptStat`을 찾아 그 객체를 `AptDetailModal`에 전달(추천 응답에 area_stats가 포함되지 않아도 안전, 원본 row에 이미 있음). dev는 `rows`를 Map으로 인덱싱해 O(1) 조회.
- **모달 상태 소유 위치**: `AptDetailModal`은 현재 `DealsTable` 내부 상태로 관리(모달 문서 §7). 추천 섹션은 별도 컴포넌트이므로 **`AiRecommendSection`이 자체 `selected` 상태를 갖고 `AptDetailModal`을 조건부 렌더**한다(DealsTable 것과 독립 인스턴스). → DealsTable 변경 불필요.
- **거래탭/지역 변경 시**: 추천 섹션도 열린 모달을 닫아야 함(§6). `AiRecommendSection` 내부 `useEffect(() => setSelected(null), [dealType, lawdCd])`.
- **예산내 하이라이트 이관(선택)**: 원본이 카드 미니표에서 하던 "예산 이하 평형 초록 표시"를 살리려면, `AptDetailModal`에 **옵션 prop `budgetMax?: number`**를 추가해 평수표 행을 하이라이트하는 확장을 고려. **단 이는 모달 스펙 변경**이므로 이번 범위 밖 → 별도 후속 제안(§8)으로 남기고, MVP 1차는 모달을 있는 그대로 재사용(하이라이트 없이). 이 결정은 dev/마스터에 명확히 전달.

### 4-D. 반응형 그리드
- 데스크톱: `grid grid-cols-2 lg:grid-cols-3 gap-4` (상위 10개면 3열×3~4행).
- 태블릿(≥640 <1024): 2열.
- 모바일(<640): **1열**(`grid-cols-1`). 카드 풀폭, 세로 스크롤. 점수 막대 3종은 모바일에서도 세로로 충분(카드 높이 여유).
- 그리드는 섹션 카드 내부 패딩 안. `mt-4`로 폼과 간격.

## 6. 상태 초기화 / 탭·지역 변경 시

- **거래탭 변경(`dealType`)**: 매매↔전세는 후보 풀(`rows`)과 "가격점수" 의미가 달라짐(§0 확인필요). → **추천 결과 폐기 + 섹션 접힘으로 복귀**. 열린 모달도 닫음. 사용자가 새 탭에서 다시 "추천 받기"를 누르게. (조용히 재계산하면 이전 탭 기준 점수가 남아 오인)
- **지역 재조회(`lawdCd` 변경)**: 결과 폐기 + 접힘 복귀 + 모달 닫기.
- **가중치 슬라이더 변경**: 폐기 아님 — 즉시 재정렬만(§3-B).
- **예산/연도 변경**: 이건 후보 필터·컷오프에 영향(외부호출 불필요하나 재필터 필요) → **"추천 받기" 재실행 유도**. 슬라이더와 달리 자동 재계산하지 않고, 값 바뀌면 "조건이 바뀌었어요 · 다시 추천 받기" 힌트 노출(변경 감지). 근거: 예산/연도는 후보 컷오프를 바꿔 결과 집합 자체가 달라지므로 명시적 재실행이 안전.
  - 구현: `running/result`에서 예산·연도·예산유형이 마지막 실행값과 다르면 CTA 라벨을 "다시 추천 받기"로 바꾸고 작은 dirty 배지.

## 7. API 응답 데이터 계약 (화면이 요구하는 필드)

> dev가 Route Handler를 만들 때 **화면이 소비할 최소 필드**. 실제 계산/캐시 구현은 dev 몫이나, 아래 필드가 없으면 화면(특히 §3-B 무호출 재정렬)이 불가하므로 계약으로 고정.

요청 params(기획 §0 계승, MVP): `lawd_cd, gu, budget, budget_type(max|min), deal_type, year, w_price, w_subway, w_new`. (평지·세대수 파라미터 없음)

응답 항목 1건(추천 단지):

```ts
interface RecommendItem {
  name: string;
  dong: string;
  build_year: string;       // meta 표시
  avg_price: number | null; // "전체 평균" 표시
  // 지하철 배지
  subway_name: string;      // 없으면 "" → 배지 미표시 or "미확인"
  subway_dist: number;      // m, 미확인 시 9999(원본 규약 계승)
  // ★ 가중치 적용 전 "원점수" — 슬라이더 무호출 재정렬의 필수 입력
  price_score: number;      // 0~100
  subway_score: number;     // 0~100
  newbuild_score: number;   // 0~100
  // score(종합)는 클라이언트가 가중치로 재계산하므로 응답에 포함해도/안 해도 무방.
  //   초기 표시용으로 서버가 기본 가중치 기준 score를 넣어주면 편의상 사용, 이후 슬라이더 변경분은 클라 재계산.
  score?: number;           // 0~100 (초기값, optional)
  // area_stats는 응답에 넣지 않아도 됨 — 모달은 부모 rows의 원본 AptStat에서 가져옴(§4-C)
}

interface RecommendResponse {
  items: RecommendItem[];        // 상위 10개(원본 컷오프 계승)
  filled: number;                // 이번 라운드까지 채운 수 (진행 배지 N)
  total_candidates: number;      // 채워야 할 총 후보 수 (진행 배지 M)
  pending: boolean;              // 아직 미채움분 남았는지(스켈레톤/더보기 여부)
  // 부분결과/콜드 진행을 SSE로 흘릴지 단발 응답+폴링일지는 dev 결정.
  // 화면은 (filled,total_candidates,pending)만 있으면 §3 상태를 그릴 수 있음.
}
```

- **핵심**: `price_score/subway_score/newbuild_score`(원점수)가 있어야 슬라이더가 카카오를 안 때리고 `score = (w_price·price + w_subway·subway + w_new·new) / (w_price+w_subway+w_new)`로 클라 재계산 가능. 이 3필드는 **필수**.
- `apt_geo` 캐시 스키마 초안은 기획 §2에 있음(`lawd_cd,name,dong → lat,lng,subway_name,subway_dist,geocoded_at`). 스키마 확정은 dev 마이그레이션 단계.
