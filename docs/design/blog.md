# 블로그 목록/상세 — 화면 구성안

- 대상: 신규 `/blog`(목록), `/blog/[slug]`(상세) — 원본 Flask 앱엔 없던 신규 화면
- 근거 문서: `docs/planning/content-strategy.md`(기획팀, MVP=`/blog/guide` 카테고리 승인)
- 계승 대상: `docs/design/dashboard.md`(색상 규칙), `docs/design/apt-detail-modal.md`(모달 톤·상태 UX), 계산기 허브(`C:\claude code webside`) `design/blog-layout-spec.md`(v1.0) + `design/blog-pagination-search-ui-spec.md`(v1.1) — 특히 v1.1 §11 "그리드→단일 컬럼 되돌림" 피드백을 이번 설계에 선반영
- 범위 밖: "관련 지역 대시보드 링크" 아키텍처(쿼리스트링 vs `/region/[code]`) — 기획안 §6 확인, 3차(`insight` 카테고리) 착수 시점에 개발팀 확인 후 재스펙. 이번 MVP(`guide`) 글은 지역 특정 데이터가 없어 해당 없음.

---

## 0. 계산기 허브 사례에서 가져오는 판단

계산기 허브는 07-15에 목록을 `max-w-3xl` 단일 컬럼으로 시작 → 글이 22편으로 늘자 07-17 `max-w-5xl` 3열 그리드로 전환했다가, **배포 당일 사용자가 직접 반려하고 단일 컬럼 리스트로 되돌렸다**(원인: 카드 폭이 좁아져 제목 `line-clamp-2`가 이른 글자 수에서 잘림 — 텍스트 위주 카드는 좁은 컬럼에서 가독성이 떨어짐이 실사용으로 확인됨).

이 프로젝트는 MVP가 카테고리 1개·글 2~3편으로 시작해 계산기 허브의 "22편" 규모와는 상황이 다르지만, **같은 실패를 반복하지 않기 위해 처음부터 단일 컬럼 리스트로 설계**한다. 글이 수십 편으로 늘어나는 시점이 오면 그때 그리드 재검토를 별도 스펙으로 남긴다(아래 1-3 참고).

## 1. 블로그 목록 페이지 (`/blog`)

### 1-1. 레이아웃 구조 (위 → 아래)

```
[Header]                              ← 대시보드 헤더 재사용 (브랜드 그라디언트)
[페이지 컨테이너 max-w-3xl mx-auto px-4 py-8]
  [H1] 블로그
  [소개문 1~2줄]  예: "실거래가를 더 잘 읽는 법을 정리합니다. 투자 조언이 아닌 참고 정보입니다."
  ──────────────────────────────
  [카테고리 필터 탭]                   ← 1-2
  ──────────────────────────────
  [글 카드 리스트]  space-y-4 세로 스택  ← 1-4
  ──────────────────────────────
  [Footer]
```

- 대시보드의 카드 표면 스타일(`bg-white rounded-xl shadow-sm p-4 md:p-5`)을 그대로 승계 — 이 사이트에 이미 확립된 카드 톤(요약 카드, 차트 카드, 목록 테이블 카드)과 통일.
- 페이지 컨테이너는 대시보드 본문 컨테이너(`max-w-[1200px]`)보다 좁은 **`max-w-3xl`(약 768px)** 을 쓴다. 이유: 대시보드는 표/차트 등 넓은 데이터 화면이라 1200px가 필요했지만, 블로그는 텍스트 읽기 화면이라 읽기 좋은 폭이 우선(계산기 허브 blog-layout-spec.md §0과 동일 근거).

### 1-2. 카테고리 필터 — 다중 카테고리 대응 탭 UI

기획안(content-strategy.md §2)이 정의한 4개 카테고리(`guide`/`insight`/`howto`/`market`) 중 MVP는 `guide` 1개만 활성이지만, **구조는 처음부터 4개 대응**으로 설계한다.

```
[전체]  [용어·제도]  [지역별 인사이트]*  [활용법]*  [월간 시황]*
                      ↑ MVP엔 아직 글이 없어 탭 자체를 렌더하지 않음(별표는 예시 표기, 실제 화면엔 없음)
```

- **탭 노출 규칙**: 카테고리 배열을 순회하며 "해당 카테고리에 글이 1편 이상 있을 때만" 탭을 렌더한다(`categoryInfo` 같은 정적 매핑 대신 실제 데이터 유무로 필터링). MVP에서는 `guide`에만 글이 있으므로 화면엔 **"전체" + "용어·제도" 2개 탭만** 보인다 — 요구사항과 일치.
- 이렇게 하면 `insight`/`howto`/`market`에 글이 채워지는 순간 별도 화면 개편 없이 탭이 자동으로 늘어난다(디자인팀 재작업 불필요).
- **탭 스타일**: 대시보드 `DealTypeTabs`(매매/전세 탭)와 동일한 시각 언어를 재사용 — 새 컴포넌트 원형을 만들지 않는다.
  ```
  <div class="flex gap-2 overflow-x-auto bg-white rounded-xl shadow-sm p-1.5 mb-5">
    <button class="whitespace-nowrap rounded-lg px-4 py-2.5 text-sm min-h-[44px] transition-colors
        data-[active=true]:bg-brand data-[active=true]:text-white data-[active=true]:font-semibold
        text-gray-600 hover:bg-[#f5f7ff]">
      전체
    </button>
    <!-- 카테고리별 반복 -->
  </div>
  ```
  - 대시보드 `DealTypeTabs`는 `flex-1`로 2등분(고정 2개)했지만, 블로그는 카테고리 수가 가변적이므로 **`overflow-x-auto` 가로 스크롤 가능한 탭 목록**으로 바꾼다(모바일에서 4개 이상 탭이 잘리지 않게). 각 탭은 `whitespace-nowrap`으로 줄바꿈 방지.
  - "전체"는 항상 첫 번째 고정, 기본 선택.
  - 활성 탭 = 브랜드 프라이머리 `#3f51b5` 배경 + 흰 텍스트(대시보드 활성 버튼/탭 규칙 그대로). 비활성 = 회색 텍스트.
  - 터치 타깃 44px 이상 확보(대시보드 원칙 승계).

### 1-3. 글 카드 (`BlogPostCard`)

- 카드 전체가 `<Link href="/blog/{slug}">`, 클릭 영역 카드 전체.
  ```
  <Link class="block bg-white rounded-xl shadow-sm p-4 md:p-5 mb-4
      hover:shadow-md transition-shadow border border-transparent hover:border-[#c5cae9]">
  ```
- 내부 구성(위→아래):
  1. **카테고리 배지** — 좌상단 단독 배치(1-4 색상 참고). `<span>` 작은 pill.
  2. **제목(H2)** — `text-base md:text-lg font-bold text-brandDark line-clamp-2 mt-2`.
     - **주의(계산기 허브 교훈 반영)**: 카드 폭을 좁히는 다열 그리드를 쓰지 않으므로 `line-clamp-2`가 계산기 허브 사례처럼 이른 글자 수에서 잘리는 문제가 원천적으로 생기지 않는다.
  3. **요약(1~2줄)** — `<p class="text-sm text-gray-500 line-clamp-2 mt-1.5">`.
  4. **메타 줄** — `<div class="flex items-center gap-2 text-xs text-gray-400 mt-2">`
     - 발행일 `<time dateTime={ISO}>2026. 7. 17.</time>`
     - 구분점 `·`
     - 읽기시간(선택) `읽는 시간 3분`

### 1-4. 반응형 — 데스크톱도 단일 컬럼 리스트 (그리드 미채택)

**데스크톱/모바일 공통으로 `space-y-4` 세로 스택 1열을 쓴다.** 0장에서 밝힌 이유(계산기 허브의 그리드→리스트 되돌림 실사용 피드백)를 이 프로젝트 시작 시점부터 선반영한 결정이다. 데스크톱에서 다열 그리드를 쓰지 않는 대신, 읽기 폭(`max-w-3xl`)을 확보해 카드 자체는 데스크톱에서 더 여유 있게 보인다(모바일보다 카드 좌우 패딩·폰트가 소폭 커짐, 5-2 참고).

| 항목 | 모바일(<640px) | 데스크톱(≥640px) |
|---|---|---|
| 카드 배치 | 1열 리스트 | 1열 리스트(그리드 아님) |
| 카드 패딩 | `p-4` | `md:p-5` |
| 제목 크기 | `text-base` | `md:text-lg` |
| 요약 크기 | `text-sm` | 동일 |

> **향후 재검토 트리거**: `guide` 외 카테고리가 열리고 전체 글이 대략 15~20편을 넘어서면, 그 시점에 "그리드 도입"을 별도 구성안으로 재검토한다. 단, 계산기 허브 사례가 보여주듯 그리드 전환은 반드시 실사용 검증(모바일 실기기 또는 최소 반응형 뷰포트 확인)을 거친 뒤 확정할 것 — 이번 문서에서 미리 그리드안을 확정하지 않는다.

### 1-5. 글 개수별 처리

| 상황 | 처리 |
|---|---|
| 0편(카테고리 선택 시 해당 카테고리에 글 없음) | 빈 상태 박스: `bg-white rounded-xl shadow-sm p-8 text-center text-sm text-gray-400 border border-dashed border-gray-200` — 문구 "아직 등록된 글이 없습니다." |
| 1~2편(MVP 초기) | 리스트에 그대로 1~2장, 추가 처리 불필요(그리드가 아니므로 "덩그러니" 문제 자체가 없음) |
| 여러 편 | 최신순(발행일 desc) 정렬 |

## 2. 블로그 상세 페이지 (`/blog/[slug]`)

### 2-1. 레이아웃 구조 (위 → 아래)

```
[Header]
[페이지 컨테이너 max-w-3xl mx-auto px-4 py-8]
  [Breadcrumb] 홈 / 블로그 / 용어·제도            ← 2-2
  [H1] 글 제목
  [메타 줄] 2026. 7. 17. · 카테고리 배지 · 읽는 시간 3분
  ──────────────────────────────
  [도입부 1~2문단]
  [면책 콜아웃 — info 또는 warning]                ← 2-3, 3장
  ──────────────────────────────
  [본문 h2/h3 + 문단 + 표]
  [(수치 인용 시) 데이터 출처 캡션]                 ← 5장
  [(insight/market류 글이면) 본문 중간 콜아웃 추가]  ← 2-3
  ──────────────────────────────
  [하단 CTA 박스]                                  ← 2-4
  [← 블로그 목록으로]                              ← 2-5
[Footer]
```

### 2-2. 브레드크럼 · 제목 · 메타

- 브레드크럼: `text-xs text-gray-400` + 링크 `hover:text-brand`. 구분자 `/`.
  `홈`(`/`) / `블로그`(`/blog`) / `{카테고리명}`(`/blog?category={slug}` 또는 카테고리 탭 상태 — 라우팅 방식은 개발팀 판단, 이 문서는 표시 텍스트만 규정).
- H1: `text-xl md:text-2xl font-bold text-brandDark mt-2`.
- 메타 줄: `mt-2 flex flex-wrap items-center gap-2 text-xs md:text-sm text-gray-500`
  - `<time dateTime>` 발행일
  - `·` 카테고리 배지(목록과 동일 배지 재사용)
  - `·` 읽는 시간 N분(선택)

### 2-3. 면책 콜아웃 위치

- **필수 위치 1**: 본문 최상단, **도입부 직후**(H1/메타 다음 문단 1~2개를 읽은 뒤, 본문 h2 소제목이 시작되기 전). 이 프로젝트는 기획안(content-strategy.md §4) 상 "본문 최상단 1개 + insight/market 카테고리는 중간 수치 인용 직후 1개 추가"가 원칙이므로, MVP(`guide`)는 **콜아웃 1개만**으로 충분하다(용어·제도 설명 글은 수치 인용이 거의 없음).
- 스타일·variant는 3장 참고.

### 2-4. 하단 CTA — "이 사이트에서 직접 조회해보기"

기획안 §6에 따라 "관련 지역 대시보드 링크"는 이번 MVP 범위 밖(고정 URL 미확정)이므로, **단순 홈(`/`) 링크 버튼**으로 충분하다.

```
<div class="bg-white rounded-xl shadow-sm p-5 mt-8 text-center">
  <p class="text-sm text-gray-600 mb-3">직접 원하는 지역의 실거래가를 조회해보세요.</p>
  <Link href="/" class="inline-flex items-center justify-center min-h-[44px] rounded-lg
      bg-brand px-6 py-2.5 text-sm font-semibold text-white hover:bg-[#303f9f] transition-colors">
    대시보드에서 조회하기 →
  </Link>
  <p class="text-xs text-gray-400 mt-3 leading-relaxed">
    이 글에 포함된 실거래가 통계는 참고용 정보이며, 투자 자문이나 시세 보증을 목적으로 하지 않습니다.
    매수·매도 등 의사결정 전 국토교통부 실거래가 공개시스템 원자료와 전문가 확인을 함께 거치시기 바랍니다.
  </p>
</div>
```

- 버튼 색·스타일은 대시보드 `LoadButton`(조회하기 버튼)과 동일 클래스 재사용 — 새 버튼 스타일 발명 안 함.
- 면책 문구는 기획안 §4의 예시 문구를 그대로 반복(문구 자체는 기획팀 산출물, 디자인은 배치·톤만 규정).
- **3차(`insight`) 이후**: 특정 지역/단지로 보내는 딥링크가 필요해지면, 이 버튼의 `href`만 교체하면 되는 구조(버튼 컴포넌트 자체는 그대로 재사용 가능) — 개발팀이 쿼리스트링/동적 라우트 중 확정하면 그때 `href` 계산 로직만 추가.

### 2-5. 목록으로 돌아가기

- CTA 박스 아래 `mt-6`: `<Link href="/blog" class="text-sm text-gray-500 hover:text-brand">← 블로그 목록으로</Link>`

## 3. 면책 콜아웃 컴포넌트 스펙 (`Callout`)

기획안 §4가 이 사이트는 계산기 허브보다 **warning 사용 빈도가 높을 것**이라 명시했으므로, warning을 info보다 시각적으로 한 단계 더 뚜렷하게 정의한다.

```tsx
interface CalloutProps {
  variant: "info" | "warning";
  children: React.ReactNode;
}
```

### 3-1. info — 일반 안내/참고

- 용도: "이 글은 참고용입니다", "용어 설명 보충" 등 경고 수준은 아닌 일반 안내.
- 색: 이 프로젝트의 "하락(파랑)" 톤을 재사용 — 새 색 도입 안 함.
  ```
  <div class="flex gap-2.5 rounded-lg border border-[#bbdefb] bg-[#e3f2fd] px-4 py-3 my-4">
    <span aria-hidden="true" class="text-[#1e88e5] text-lg leading-none">ℹ️</span>
    <p class="text-sm text-[#0d47a1] leading-relaxed">{children}</p>
  </div>
  ```
  - 배경 `#e3f2fd`(대시보드 하락 배지 배경과 동일 토큰), 아이콘/포인트 색 `#1e88e5`(하락 텍스트 색과 동일), 본문은 조금 더 짙은 `#0d47a1`(가독성 위해 배경 위 대비 확보, 신규지만 같은 블루 계열).

### 3-2. warning — 투자자문 오인 방지 / 데이터 한계 고지 (사용 빈도 높음)

- 용도: "투자 자문이 아닙니다", "최근 달 데이터는 신고 지연으로 적을 수 있습니다" 등 **반드시 눈에 띄어야 하는** 면책·주의.
- 색: 대시보드의 "재건축 임박(rebuild-ready)" 배지 색 계열(`#c62828`/`#fce4ec`)을 재사용 — "상승(#e53935)"과는 의도적으로 구분한다. 이유: 본문에 실거래가 상승/하락 수치가 함께 등장하는 글이 많은데(`text-up: #e53935`), 경고 박스에 같은 빨강을 쓰면 "가격이 올랐다"는 의미와 "투자자문 아님" 경고가 시각적으로 섞여 혼동을 줄 수 있다. `rebuild-ready`는 이미 이 사이트에서 "가장 강한 주의가 필요한 상태"를 나타내는 색으로 확립돼 있어 그 의미를 그대로 승계한다.
  ```
  <div class="flex gap-2.5 rounded-lg border-l-4 border-[#c62828] bg-[#fce4ec] px-4 py-3 my-4">
    <span aria-hidden="true" class="text-[#c62828] text-lg leading-none">⚠️</span>
    <p class="text-sm text-[#7f1d1d] leading-relaxed font-medium">{children}</p>
  </div>
  ```
  - **info와의 차이**: 좌측 `border-l-4` 강조 바(info엔 없음) + 텍스트 `font-medium`(info는 일반 굵기) — warning이 시각적으로 한 단계 더 강해 보이게.
  - **pulse 애니메이션은 쓰지 않는다**: 대시보드의 `rebuild-ready` pulse는 "재건축 임박"이라는 실시간성 강조 용도였고, 정적 텍스트 콘텐츠인 블로그 면책 문구에 애니메이션을 쓰면 과하다(오히려 주의를 분산). 색만 승계하고 애니메이션은 제외.

### 3-3. 공통 규칙

- 아이콘은 `aria-hidden="true"`로 감싸 스크린리더가 이모지를 중복 낭독하지 않게 한다.
- 콜아웃은 항상 `<p>` 또는 짧은 리스트만 담는다 — 콜아웃 안에 표/차트 등 복잡한 구조 금지(계산기 허브 `callout` 패턴과 동일 원칙).
- 모바일/데스크톱 반응형 차이 없음(패딩·폰트 고정, 텍스트 자체가 짧아 반응형 분기 불필요).

## 4. 카테고리 배지 색상 (4개 팔레트)

기획안 §2의 4개 카테고리에 각각 배지 색을 지정한다. 브랜드 프라이머리(`#3f51b5`)·상승/하락(`#e53935`/`#1e88e5`)·재건축 경고(`#c62828`/`#e65100`)와 충돌하지 않는 새 계열만 선택했다.

| 카테고리 | 이름 | 배지 텍스트색 | 배지 배경 | 비고 |
|---|---|---|---|---|
| `guide` | 용어·제도 | `#3f51b5`(브랜드 프라이머리) | `#e8eaf6` | MVP 활성 카테고리. 브랜드 컬러를 그대로 써서 "가장 기본/안전한 카테고리"라는 인상을 준다. |
| `insight` | 지역별 인사이트 | `#00897b`(틸) | `#e0f2f1` | 데이터 분석 느낌의 차분한 청록. 상승/하락 빨강·파랑과 명확히 구분. |
| `howto` | 활용법 | `#f57f17`(호박색) | `#fff8e1` | 대시보드 재건축 배지 주황(`#e65100`)과는 톤을 달리한(더 밝은 호박색) 별개 색 — 재건축 경고와 혼동 방지. |
| `market` | 월간 시황 | `#6a1b9a`(퍼플) | `#f3e5f5` | 4개 중 유일하게 기존 팔레트에 없던 계열. YMYL 위험도가 가장 높은 카테고리라 시각적으로도 가장 눈에 띄게 구분해, 작성/검수 단계에서 "이 카테고리는 특히 조심"이라는 신호를 편집자에게도 준다. |

- 배지 공통 마크업: `<span class="inline-block rounded-full px-2.5 py-0.5 text-xs font-semibold" style="color:{텍스트색}; background:{배경색}">{카테고리명}</span>`
- MVP는 `guide` 1개만 실제로 노출되지만, 위 4색은 지금 전부 확정해 `insight`/`howto`/`market` 오픈 시 추가 디자인 작업 없이 바로 적용 가능하게 한다.

## 5. 데이터 출처 표기 영역

기획안 §4 "모든 수치 인용 글은 조회 기준일 표기 필수" 원칙을 화면 스타일로 구체화한다.

### 5-1. 본문 문단 내 수치 인용

- 수치가 포함된 문단 바로 아래에 작은 캡션 한 줄:
  ```
  <p class="text-xs text-gray-400 mt-1">국토교통부 실거래가 공개시스템 공공데이터 기반, 조회 기준일 2026-07-17</p>
  ```
- 위치: 해당 문단 **직후**(문단과 캡션 사이 별도 여백 없이 붙여서, 캡션이 "이 문단 수치의 출처"임을 시각적으로 명확히 함). 캡션과 다음 문단 사이엔 일반 문단 간격(`mt-4`) 유지.
- 같은 섹션(h2 하나) 안에서 같은 기준일 수치를 여러 번 인용하면, **첫 인용에만 캡션을 달고 이후는 생략 가능**(과도한 반복 방지) — 단, 표가 별도로 나오면 5-2 규칙을 표마다 따로 적용.

### 5-2. 표 안 수치 인용

- 표 하단에 캡션 형태로 배치(표 자체 안이 아니라 표 바로 아래):
  ```
  <table class="w-full text-sm border-collapse my-2"> ... </table>
  <p class="text-xs text-gray-400 mt-1.5">국토교통부 실거래가 공개시스템 공공데이터 기반, 조회 기준일 2026-07-17</p>
  ```
- 시맨틱하게는 `<caption>` 태그도 고려할 수 있으나, `<caption>`은 브라우저 기본 렌더 위치가 표 상단이라(스타일로 하단 이동 가능하지만) 이번 스펙은 **표 바로 아래 별도 `<p>`** 방식을 권장(계산기 허브 표 기반 콘텐츠에서도 별도 `<p>` 캡션 방식이 더 흔함 — 마크업 단순함 우선).

### 5-3. 공통 스타일 규칙

- 캡션 스타일은 항상 `text-xs text-gray-400`(대시보드 라벨 텍스트 톤 `text-gray-400`과 동일 — 새 회색 도입 안 함), 이탤릭·굵기 변화 없음(본문과 명확히 구분되되 과하게 튀지 않도록).
- 기준일은 **글 발행일이 아니라 "그 수치를 실제로 조회한 날짜"** — 기획팀이 콘텐츠 데이터와 함께 제공(디자인 범위 밖, 표기 스타일만 규정).
- 외부 통계(언론 보도 등) 인용 자체가 기획안상 금지이므로, 이 캡션 포맷은 **항상 "국토교통부 실거래가 공개시스템"** 한 출처만 표기한다(출처명이 바뀌는 케이스는 이번 스펙에서 다루지 않음).

## 6. 다른 팀 전달 사항

**개발팀**
- `BlogPostCard`, `CategoryTabs`, `Callout(info/warning)` 3개 신규 컴포넌트 필요. 색상 값은 4장/3장 표 그대로 하드코딩하거나 Tailwind 커스텀 컬러로 등록(대시보드 `tailwind.config` 확장 패턴 재사용).
- 카테고리 탭은 "글이 있는 카테고리만 렌더" 규칙(1-2) — 정적 4개 배열이 아니라 실제 데이터 유무로 필터링할 것.
- 하단 CTA(2-4) 버튼 `href="/"` 고정 — 3차(`insight`) 착수 전까지는 로직 변경 불필요.

**기획팀 확인/제공 요청**
- 목록 소개문·카테고리별 브레드크럼 표시명·각 글의 요약(2줄 이내)·읽기시간·수치 인용 시 조회 기준일 값.
- `guide` 카테고리 브레드크럼 표시명 확정("용어·제도" 가안 — 기획안 §2 "이름" 컬럼과 일치하는지 재확인 요청).

**마스터 확인 필요**
- 없음 — 신규 색은 기존 팔레트(브랜드/상승/하락/재건축 경고)와 겹치지 않는 3색(틸/호박/퍼플)만 추가했고, 4장에서 근거를 남겼다. 이견 있으면 회수 가능.
